# Часть 1. Что вообще считается «ресурсом» 📋

🧠 **Ресурс** — это всё, что нужно сначала **взять** (выделить), а потом обязательно **вернуть** (освободить). Если не вернуть — будет беда: утечка, зависшая блокировка, не закрытый файл.

| Ресурс | Взять (выделить) | Вернуть (освободить) |
|---|---|---|
| Память | `new` / `malloc` | `delete` / `free` |
| Файл | `fopen` / открыть | `fclose` / закрыть |
| Сетевой сокет | создать/подключить | закрыть |
| Блокировка мьютекса | `lock` | `unlock` |
| Дескриптор ОС | `HANDLE` (Win), `fd` (Linux) | закрыть |
| 🔧 Подъёмник в гараже | поднять | опустить |
| 🔧 Станок | включить | выключить |

🤖 У каждой строки — **пара**: взял → верни. RAII делает так, чтобы «верни» происходило **само**, автоматически, и его невозможно было забыть.

# Часть 2. Проблема: освобождать вручную — ненадёжно ⚠️

🧠 Если освобождать ресурс «руками» в конце функции, очень легко промахнуться — особенно если в середине есть `return` (или случится ошибка).

```cpp
#include <cstdio>

// ⚠️ ПЛОХО: закрываем файл вручную
void plokho(bool oshibka) {
    std::FILE* f = std::fopen("log.txt", "w");   // взяли ресурс
    if (oshibka) {
        return;                 // 💥 вышли и ЗАБЫЛИ fclose(f) -> файл остался открыт (утечка!)
    }
    std::fclose(f);             // закрываем только если дошли сюда
}
```

🔧 Образ из гаража: поднял машину на **подъёмнике**, отвлёкся, что-то пошло не так — ушёл, а подъёмник остался поднятым. Опасно. «Не забудь опустить в конце» — ненадёжное правило: человек забывает.

🤖 Нужен способ, при котором «верни ресурс» происходит **автоматически на выходе** — что бы ни случилось. Это и есть RAII.

# Часть 3. Решение: своя RAII-обёртка 🛡️

🧠 Делаем класс-«хозяина» ресурса: **конструктор выделяет**, **деструктор освобождает**. Деструктор вызывается автоматически на выходе из `{ }` (помнишь Урок A.4 и лампу из Части C?).

## Урок 3.1. Простой пример — подъёмник

```cpp
#include <iostream>

class Podyemnik {                 // ресурс — «поднятое» состояние
public:
    Podyemnik()  { std::cout << "Podyemnik PODNYAT\n"; }     // ВЗЯЛИ ресурс
    ~Podyemnik() { std::cout << "Podyemnik OPUSCHEN\n"; }    // ВЕРНУЛИ ресурс
    void rabota() const { std::cout << "  remontiruem mashinu\n"; }
};

void osmotr() {
    Podyemnik p;        // подняли
    p.rabota();
}                       // вышли -> деструктор -> опустили САМ

int main() {
    osmotr();
}
```

**Ожидаемый вывод:**
```
Podyemnik PODNYAT
  remontiruem mashinu
Podyemnik OPUSCHEN
```

🤖 Мы **нигде не писали «опусти подъёмник»** — он опустился сам, в деструкторе. И опустится, даже если в `osmotr` будет `return` посередине или случится ошибка. Забыть невозможно.

## Урок 3.2. Реальный пример — обёртка над файлом

🧠 Теперь то же самое, но с **настоящим** ресурсом — файлом. Конструктор открывает, деструктор закрывает.

```cpp
#include <cstdio>      // FILE*, fopen, fclose, fprintf
#include <iostream>

class FaylLog {
public:
    FaylLog(const char* imya) {
        f_ = std::fopen(imya, "w");      // ВЫДЕЛЯЕМ ресурс: открыли файл
        std::cout << "fayl otkryt\n";
    }
    ~FaylLog() {
        if (f_) std::fclose(f_);         // ОСВОБОЖДАЕМ ресурс: закрыли файл
        std::cout << "fayl zakryt\n";
    }
    void zapisat(const char* tekst) {
        if (f_) std::fprintf(f_, "%s\n", tekst);
    }
private:
    std::FILE* f_ = nullptr;             // сам ресурс (дескриптор файла)
};

int main() {
    FaylLog log{"log.txt"};
    log.zapisat("scan started");
    log.zapisat("port 80 open");
}   // файл закроется САМ — даже если что-то пойдёт не так
```

**Ожидаемый вывод:**
```
fayl otkryt
fayl zakryt
```
А рядом появится файл `log.txt` с двумя строчками внутри — **открой и проверь**.

🤖 Важные детали «как правильно»:
- ресурс (`f_`) храним **внутри** (`private`), наружу даём только удобные методы;
- в деструкторе **проверяем `if (f_)`** — вдруг ресурс не выделился;
- деструктор **не должен бросать исключения** (просто тихо прибирается).

# Часть 4. ⚠️ Главная ловушка: копирование (Правило трёх/пяти)

🧠 Вот тут RAII чаще всего делают **неправильно**. Что будет, если такой объект **скопировать**?

```cpp
FaylLog a{"log.txt"};
FaylLog b = a;          // ⚠️ скопировался указатель f_ -> ДВА объекта на ОДИН файл!
```

🤖 По умолчанию копия просто копирует поле `f_` — теперь `a` и `b` держат **один и тот же** файл. А в конце **оба** деструктора вызовут `fclose(f_)` — файл закроется **дважды**. Это **двойное освобождение** (*double-free* / *double-close*) — серьёзная ошибка, программа может упасть.

🔧 Образ: два механика думают, что каждый владеет подъёмником, и оба пытаются его «сдать». Путаница и поломка.

✅ **Правило трёх/пяти:** если класс **владеет ресурсом** (у него есть деструктор, который что-то освобождает), то нужно решить и про **копирование**. Самый простой правильный путь для новичка — **запретить копирование**:

```cpp
class FaylLog {
public:
    FaylLog(const char* imya) { f_ = std::fopen(imya, "w"); }
    ~FaylLog() { if (f_) std::fclose(f_); }

    FaylLog(const FaylLog&)            = delete;   // запретить копирование
    FaylLog& operator=(const FaylLog&) = delete;
private:
    std::FILE* f_ = nullptr;
};
```

🤖 Теперь `FaylLog b = a;` **не скомпилируется** — ошибка поймана заранее, бага нет. Именно так устроен `std::unique_ptr`: его **нельзя копировать**, только **передавать владение** (move). Передача владения (move) — это «следующий шаг»; разберём подробно, когда будем дорабатывать.

⚠️ Запомни связь: **есть деструктор, освобождающий ресурс → подумай про копирование** (запрети его или сделай move). Иначе — двойное освобождение.

# Часть 5. ✅ Лучшее правило — Правило нуля

🧠 А самый правильный совет: **по возможности вообще не пиши свой деструктор**. Вместо «сырых» ресурсов используй **готовые RAII-типы**, и тогда тебе не нужны ни деструктор, ни возня с копированием — всё уже сделано правильно за тебя.

| Вместо «сырого» ресурса | Бери готовое RAII |
|---|---|
| `new` / `delete` | `std::unique_ptr` / `std::make_unique` |
| `new[]` массив | `std::vector` |
| `FILE*` + `fopen`/`fclose` | `std::fstream` (сам закроется) |
| `lock` / `unlock` мьютекса | `std::lock_guard` |
| строка-буфер | `std::string` |

🤖 Это называют **Правило нуля**: строй классы так, чтобы их **поля** были уже-RAII-типами (vector, string, unique_ptr…). Тогда твой класс **не нуждается** ни в деструкторе, ни в копировании — всё делают его поля. Меньше своего кода → меньше багов.

✅ **Итог-иерархия правильности:**
1. 🥇 **Правило нуля** — используй готовые RAII-типы, не пиши деструктор. *Так — почти всегда.*
2. 🥈 Если уж владеешь «сырым» ресурсом — напиши **одну** маленькую RAII-обёртку и **запрети копирование** (`= delete`).
3. 🥉 Нужна передача владения — добавь move (Правило пяти). *Это уже продвинуто.*

# Часть 6. Почему RAII спасает даже при ошибках 🛟

🧠 Главная суперсила RAII: деструктор вызывается **в любом случае** на выходе из блока — даже если программа «выскочила» из функции из-за ошибки. Поэтому ресурс освободится, даже когда обычный «`fclose` в конце» был бы **пропущен**.

🤖 Проще говоря: с ручным освобождением ошибка посередине = пропущенная уборка = утечка. С RAII — уборка происходит **всегда**. Этот «подъём с уборкой» при ошибке называется **раскрутка стека** — подробно (с запускаемым примером) в файле `Курс_C++_Move_и_раскрутка_стека.md`.

# Часть 7. Scope guard — уборка «на лету» 🧷

🧠 Иногда ресурс приходит из **C-библиотеки или системного API**, для которого **нет готовой RAII-обёртки**, а писать целый класс ради одного места — перебор. Тут выручает **scope guard** — крошечный помощник, который выполнит **любую уборку** (лямбду), когда выйдет из своей области видимости `{ }`.

🔧 **Жизненная техническая аналогия:** **пружинный рычаг безопасности** — «кнопка мёртвого человека» на газонокосилке или беговой дорожке. Пока держишь рычаг — машина работает; **отпустил** (ушёл) — она **сама** выключается. Scope guard такой же: «отпустил» = вышел из блока → уборка срабатывает сама, что бы ни случилось.

🔧 Ещё образ: **самозакрывающаяся дверь на пружине** — как ни уходи, она закроется за тобой сама.

## Урок 7.1. Как он устроен

🧠 Scope guard — это просто RAII-объект, который хранит **функцию-уборку** и вызывает её в деструкторе. Минимальная реализация:

```cpp
#include <iostream>
#include <functional>

class ScopeGuard {
public:
    ScopeGuard(std::function<void()> deystvie) : deystvie_{deystvie} {}
    ~ScopeGuard() { deystvie_(); }              // на выходе — выполнить уборку
    ScopeGuard(const ScopeGuard&)            = delete;   // не копируется (Часть 4!)
    ScopeGuard& operator=(const ScopeGuard&) = delete;
private:
    std::function<void()> deystvie_;
};

int main() {
    std::cout << "vhodim\n";
    ScopeGuard guard{[]{ std::cout << "ubiraem za soboy!\n"; }};
    std::cout << "rabotaem\n";
}   // здесь guard уничтожается -> вызывает лямбду-уборку
```

**Ожидаемый вывод:**
```
vhodim
rabotaem
ubiraem za soboy!
```

🤖 `guard` хранит лямбду; на выходе из `main` его деструктор её вызывает. Заметь `= delete` — тот самый запрет копирования из Части 4 (владелец уборки не должен копироваться).

## Урок 7.2. Где он реально нужен

🧠 Scope guard выручает, когда:
- ресурс из **C/системного API**, готовой RAII-обёртки нет;
- уборка **разовая**, и писать целый класс — лишнее;
- хочется держать «взял» и «верни» **рядом**, чтобы не забыть.

```cpp
#include <cstdio>
std::FILE* f = std::fopen("log.txt", "w");
ScopeGuard zakroem{[&]{ if (f) std::fclose(f); }};   // закроется само на выходе

std::fprintf(f, "scan started\n");
// ...даже при раннем return или ошибке файл закроется — спасибо scope guard
```

🤖 Уборка (`fclose`) написана **прямо рядом** с тем, что её требует (`fopen`) — это удобно читать и невозможно «потерять».

## Урок 7.3. Бывают три вида (на будущее)

🤖 В «больших» библиотеках есть три варианта (в стандарт C++ идут как `std::experimental::scope_exit`):
- **scope_exit** — уборка **всегда** (как у нас);
- **scope_success** — только если вышли **без ошибки**;
- **scope_fail** — только при **ошибке** (для «отката» изменений).

Подробно — когда будем дорабатывать.

✅ **Правило Боба:** есть готовый RAII-тип — бери его (Правило нуля, Часть 5). Нет, а уборка разовая — **scope guard** с лямбдой: держит «взял» и «верни» рядом и срабатывает на **любом** выходе из блока.

🎯 **Попробуй сам:** оберни в scope guard «поднял подъёмник → опусти» (просто печатай сообщения) и проверь, что «опусти» срабатывает само в конце блока — даже если добавишь `return` посередине.

## ✅ Шпаргалка «правильный RAII»

| Правило |
|---|
| Ресурс **выделяй в конструкторе**, **освобождай в деструкторе** |
| Сам ресурс держи в `private`; наружу — только методы |
| В деструкторе проверяй, что ресурс есть (`if (f_)`), и **не бросай исключений** |
| Владеешь сырым ресурсом → **запрети копирование** (`= delete`) или сделай move (Правило трёх/пяти) |
| 🥇 Лучше всего — **Правило нуля**: бери готовые RAII-типы, деструктор не пиши вовсе |

🕵️ **Реверс-связь:** реверсер ищет в коде **пары** «взял–вернул»: `fopen`/`fclose`, `new`/`delete`, `lock`/`unlock`. Если «вернул» где-то пропущен — это утечка/баг, который замечают и атакующие, и защитники.

## 📖 Словарик

| Слово | Простыми словами |
|---|---|
| **Ресурс** | То, что надо взять и обязательно вернуть |
| **RAII-обёртка** | Класс: ctor выделяет ресурс, dtor освобождает |
| **Двойное освобождение** | Один ресурс вернули дважды → ошибка |
| **`= delete`** | Запретить операцию (например, копирование) |
| **Правило трёх/пяти** | Есть деструктор-владелец → реши и про копирование/move |
| **Правило нуля** | Используй готовые RAII-типы, не пиши деструктор |
| **Move (передача владения)** | Отдать ресурс другому объекту, не копируя |
| **Scope guard** | RAII-помощник: выполняет уборку-лямбду на выходе из блока |

---

> 🤖 Главное: **выделил в конструкторе → освободил в деструкторе**, ресурс держи внутри, в деструкторе проверяй и не бросай. А чтобы было **правильно**: владеешь сырым ресурсом — запрети копирование; но лучше всего — **Правило нуля** (готовые RAII-типы, без своего деструктора).
