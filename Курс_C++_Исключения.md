# Часть 1. Зачем нужны исключения

🧠 Иногда функция **физически не может** сделать то, что просят: поделить на ноль, открыть несуществующий файл, взять элемент за границей списка. Как об этом сообщить?

**Старый способ — код ошибки:** функция возвращает `-1` или `false`, а ты после каждого вызова проверяешь. Беда: **легко забыть проверить**, и код засоряется проверками.

**Исключения** — функция «кидает» ошибку, и та **сама перелетает вверх** к тому месту, где её ловят, минуя всё между. Тихо проигнорировать ошибку нельзя.

🔧 **Аналогия:** **аварийная кнопка** на станке. Случилась беда — нажал кнопку (`throw`), станок мгновенно останавливается, а сигнал летит прямо к оператору (`catch`), минуя все обычные шаги. Оператор решает, что делать.

# Часть 2. Три слова: `throw`, `try`, `catch`

🧠 Весь механизм — это три слова:
- **`throw`** — «кинуть» ошибку (нажать аварийную кнопку);
- **`try { ... }`** — «попробуй выполнить этот блок»;
- **`catch (...) { ... }`** — «поймай ошибку, если она случится в `try`».

```cpp
#include <iostream>
#include <stdexcept>

int divide(int a, int b) {
    if (b == 0) {
        throw std::runtime_error("division by zero!");   // не можем -> кидаем ошибку
    }
    return a / b;
}

int main() {
    try {
        std::cout << divide(10, 2) << "\n";   // 5
        std::cout << divide(7, 0) << "\n";    // тут вылетит ошибка
        std::cout << "this line won't run\n";
    } catch (const std::exception& e) {
        std::cout << "Caught an error: " << e.what() << "\n";
    }
    std::cout << "Program keeps running\n";
}
```

**Ожидаемый вывод:**
```
5
Caught an error: division by zero!
Program keeps running
```

🤖 Что произошло:
- `divide(10, 2)` отработал нормально → `5`;
- `divide(7, 0)` нажал «аварийную кнопку» (`throw`) → выполнение **выпрыгнуло** из `try`, строчка `"this line won't run"` **пропущена**;
- управление прыгнуло в `catch`, там напечатали сообщение;
- после `try/catch` программа **спокойно продолжила** работу (не упала!).

`e.what()` — это текст ошибки, который мы задали в `throw`.

# Часть 3. Что бросать и что ловить

🧠 По правилам кидают **объекты-исключения** из стандартной библиотеки (заголовок `<stdexcept>`):
- `std::runtime_error` — ошибка во время работы (как у нас);
- `std::out_of_range` — выход за границу;
- `std::invalid_argument` — плохой аргумент; и другие.

🤖 Все они — **потомки** `std::exception` (помнишь наследование из [`Курс_C++_ООП_продвинуто.md`](Курс_C++_ООП_продвинуто.md)?). Поэтому одна строчка ловит **любую** из них:

```cpp
catch (const std::exception& e) { ... }   // поймает и runtime_error, и out_of_range, и …
```

### Семейство стандартных ошибок (дерево)

🧠 Стандартные исключения образуют **дерево** (через наследование):

```
std::exception                     ← общий предок ВСЕХ
├── std::logic_error               ← «ошибка в логике» = по сути БАГ программиста
│   ├── std::invalid_argument      ← плохой аргумент
│   ├── std::out_of_range          ← индекс/ключ за границей
│   ├── std::length_error          ← запрошен слишком большой размер
│   └── std::domain_error          ← значение вне допустимой области
└── std::runtime_error             ← «беда во время работы», заранее не угадать
    ├── std::range_error           ← результат вне диапазона
    ├── std::overflow_error        ← переполнение при вычислении
    └── std::underflow_error       ← «антипереполнение»
(отдельно) std::bad_alloc          ← не хватило памяти (бросает сам `new`)
```

🤖 Две главные ветки — и смысл у них **разный**:
- **`logic_error`** — это **баг**: программист должен был не допустить (передал плохой аргумент, полез за границу). По-хорошему лечится в коде, а не «ловится».
- **`runtime_error`** — **внешняя беда**, которую заранее не угадать (нет памяти, сбой системы, данные не влезли). Вот её и обрабатывают `try/catch`.

### Каталог: какой готовый тип когда кидать

| Тип (заголовок) | Когда кидать | Пример `throw` |
|---|---|---|
| `std::invalid_argument` `<stdexcept>` | плохое значение аргумента | `throw std::invalid_argument("port must be 1..65535");` |
| `std::out_of_range` `<stdexcept>` | индекс/ключ за границей | `throw std::out_of_range("index too big");` |
| `std::length_error` `<stdexcept>` | запрошен слишком большой размер | `throw std::length_error("too many items");` |
| `std::domain_error` `<stdexcept>` | значение вне области функции | `throw std::domain_error("sqrt of negative");` |
| `std::runtime_error` `<stdexcept>` | общая ошибка времени выполнения | `throw std::runtime_error("connection failed");` |
| `std::overflow_error` `<stdexcept>` | переполнение при вычислении | `throw std::overflow_error("number too big");` |
| `std::bad_alloc` `<new>` | кончилась память | обычно **ловишь**, а не кидаешь (кидает `new`) |

⚠️ Почти все берут **текст-сообщение** в конструкторе (`const std::string&` / `const char*`). Особняком — `std::bad_alloc`: у него сообщения нет, руками его не кидают (его бросает оператор `new`, а ты можешь поймать).

### Пример: кидаем готовые типы по ситуации

🛠 Функция-проверка кидает **разные стандартные** исключения в зависимости от проблемы:

```cpp
#include <iostream>
#include <stdexcept>
#include <string>
#include <vector>

// вернуть имя сервиса по номеру порта, проверяя ввод
std::string serviceName(const std::vector<std::string>& names, int port) {
    if (port < 1 || port > 65535)
        throw std::invalid_argument("port must be 1..65535");   // плохой аргумент

    if (port >= (int)names.size())
        throw std::out_of_range("no name for this port");       // за границей списка

    return names.at(port);
}

int main() {
    std::vector<std::string> names = {"reserved", "tcpmux", "echo"};
    try {
        std::cout << serviceName(names, 70000) << "\n";   // 70000 -> invalid_argument
    }
    catch (const std::invalid_argument& e) {
        std::cout << "Bad argument: " << e.what() << "\n";
    }
    catch (const std::out_of_range& e) {
        std::cout << "Out of range: " << e.what() << "\n";
    }
}
```

**Ожидаемый вывод:**
```
Bad argument: port must be 1..65535
```

🤖 Идея: **каждой беде — свой готовый тип**. Ловящий код по типу сразу понимает, **что** случилось, и реагирует точно (`invalid_argument` — поправь ввод; `out_of_range` — проверь размер). Не нужно разбирать текст сообщения — достаточно типа.

### А ещё их кидает и сама библиотека

🛠 Помнишь `.at()` у вектора — он проверяет границы? Если выйти за край, он **сам бросает** `std::out_of_range`:

```cpp
#include <iostream>
#include <vector>

int main() {
    std::vector<int> v = {10, 20, 30};
    try {
        std::cout << v.at(5) << "\n";          // ⛔ элемента №5 нет -> бросок!
    } catch (const std::out_of_range& e) {
        std::cout << "Out of range: " << e.what() << "\n";
    }
}
```

🤖 Вот почему `.at(i)` безопаснее, чем `[i]`: при выходе за границу он **честно бросает ошибку**, а не лезет в чужую память.

🧠 Можно ловить по-разному:
- `catch (const std::out_of_range& e)` — **конкретную** ошибку;
- `catch (const std::exception& e)` — **любую** стандартную (базовый класс);
- `catch (...)` — вообще **что угодно** (на крайний случай).

⚠️ Порядок важен: сначала пиши **конкретные** `catch`, потом **общие** (`catch` проверяются сверху вниз, как `else if`).

# Часть 4. Свои типы исключений 🏷️

🧠 Стандартных ошибок (`runtime_error`, `out_of_range`…) хватает не всегда. Часто хочется **свой** тип ошибки, который:
- понятно называется (`NetworkError`, а не безликий `runtime_error`);
- **несёт данные** об ошибке (какой хост, какой порт, какой код) — а не только текст.

## Урок 4.1. Создаём свой тип

🧠 Проще всего — **унаследовать** свой класс от `std::runtime_error` (а тот — потомок `std::exception`). Тогда бесплатно получаем `.what()`, и наш тип ловится и как `std::exception`.

```cpp
#include <stdexcept>
#include <string>

// свой тип ошибки: наследуем от std::runtime_error
class NetworkError : public std::runtime_error {
public:
    NetworkError(std::string host, int port)
        : std::runtime_error("cannot connect to " + host + ":" + std::to_string(port)),
          host_(host), port_(port) {}        // текст -> в базу, свои данные -> в поля

    const std::string& host() const { return host_; }   // доступ к доп. данным
    int port() const { return port_; }
private:
    std::string host_;
    int port_;
};
```

🤖 Разбор:
- `: public std::runtime_error` — наследуемся (помнишь наследование из [`Курс_C++_ООП_продвинуто.md`](Курс_C++_ООП_продвинуто.md)?).
- В список инициализации передаём текст в **базовый** конструктор `std::runtime_error(...)` — он и станет `.what()`.
- Плюс храним **свои поля** (`host_`, `port_`) — то, чего у стандартных исключений нет.

## Урок 4.2. Бросаем и ловим ИМЕННО свой тип

```cpp
#include <iostream>

void connectTo(const std::string& host, int port) {
    // ...тут была попытка подключиться и не вышло...
    throw NetworkError(host, port);          // кидаем СВОЙ объект-ошибку
}

int main() {
    try {
        connectTo("example.com", 443);
    }
    catch (const NetworkError& e) {          // ловим ИМЕННО свой тип
        std::cout << "Network error: " << e.what() << "\n";
        std::cout << "  host = " << e.host() << ", port = " << e.port() << "\n";
    }
    catch (const std::exception& e) {        // всё остальное — общий запасной
        std::cout << "Other error: " << e.what() << "\n";
    }
}
```

**Ожидаемый вывод:**
```
Network error: cannot connect to example.com:443
  host = example.com, port = 443
```

🤖 Вот ради чего всё затевалось: в `catch (const NetworkError& e)` мы не просто прочитали текст, а **достали структурированные данные** — `e.host()`, `e.port()`. С обычной строкой так не выйдет. Свой тип = ошибка несёт **полезную информацию**, которую удобно обработать (например, переподключиться именно к этому хосту).

⚠️ Порядок `catch` важен: **сначала конкретный** `NetworkError`, **потом общий** `std::exception`. Поставишь наоборот — общий перехватит первым, и до конкретного дело не дойдёт (компилятор обычно даже предупредит).

## Урок 4.3. Семейство своих ошибок (иерархия)

🧠 Можно сделать **свой базовый** класс ошибки, а от него — конкретные. Тогда одним `catch` по базе ловишь **все свои** ошибки сразу.

```cpp
class AppError : public std::runtime_error {
    using std::runtime_error::runtime_error;     // взять конструкторы базы
};

class FileError : public AppError { using AppError::AppError; };
class DbError   : public AppError { using AppError::AppError; };

// ...
try {
    // ...что-то, что может кинуть FileError или DbError...
    throw FileError("file not found");
}
catch (const AppError& e) {        // ловит ЛЮБУЮ нашу ошибку (и File, и Db)
    std::cout << "App error: " << e.what() << "\n";
}
```

🤖 `using std::runtime_error::runtime_error;` — «возьми конструкторы базы», чтобы не переписывать их. Теперь `throw FileError("...")` работает, а `catch (const AppError&)` ловит **всё семейство**. Это как с транспортом: поймал «любой `Vehicle`» — поймал и `Car`, и `Motorcycle`.

## Урок 4.4. Повторный бросок: `throw;`

🧠 Иногда в `catch` хочется что-то сделать (например, записать в лог) и **передать ошибку дальше** — пусть разбирается тот, кто выше. Для этого — **`throw;`** без аргумента:

```cpp
try {
    connectTo("example.com", 443);
}
catch (const NetworkError& e) {
    std::cout << "log: " << e.what() << "\n";   // частично обработали (записали в лог)
    throw;                                       // пере-бросили ТУ ЖЕ ошибку дальше
}
```

⚠️ Пиши именно `throw;` (пустой), а не `throw e;`. Пустой сохраняет **настоящий тип** исключения; `throw e;` может «обрезать» его до базового типа (это называют **срезкой**, *slicing*).

> 📌 Минимальный свой тип можно сделать и прямо от `std::exception`, переопределив `what()`:
> ```cpp
> class MyError : public std::exception {
> public:
>     const char* what() const noexcept override { return "my custom error"; }
> };
> ```
> Но наследоваться от `std::runtime_error` обычно удобнее — он сам хранит текст.

✅ **Правило Боба:** нужен понятный тип ошибки с **данными** — сделай свой класс-наследник `std::runtime_error`. Лови **сначала конкретные** типы, потом общий `std::exception`. Хочешь обработать частично и пустить дальше — `throw;`.

🎯 **Попробуй сам:** сделай свой тип `PortScanError` (с полями `host` и `port`), кидай его в функции-«сканере», а в `main` поймай его **отдельным** `catch`, выведя host и port.

# Часть 5. Исключения + RAII = безопасно 🛡️

🧠 Важнейшая связь. Когда исключение летит вверх, по пути **срабатывают деструкторы** локальных объектов — это **раскрутка стека** (подробно в [`Курс_C++_Move_и_раскрутка_стека.md`](Курс_C++_Move_и_раскрутка_стека.md)). Значит, все RAII-объекты (`unique_ptr`, `string`, `QFile`, scope guard…) **сами приберутся**, даже когда случилась ошибка.

🤖 Отсюда правильное разделение:
- **RAII** отвечает за **уборку** (закрыть файл, вернуть память) — само, всегда;
- **`try/catch`** отвечает за **реакцию** на ошибку (что показать, что сделать дальше).

✅ **Правило Боба:** не пиши `try/catch` только ради того, чтобы что-то «закрыть» при ошибке — это сделает **RAII**. `try/catch` нужен там, где ты вправду хочешь **обработать** ошибку.

# Часть 6. Когда использовать (и когда нет)

🧠 Исключения — для **по-настоящему исключительных** ситуаций, которые нельзя решить на месте: файла нет, сеть отвалилась, кончилась память, пользователь ввёл чушь.

⚠️ **Не используй** исключения как обычный цикл/развилку. Они для **ошибок**, а не для нормального хода программы («нашёл/не нашёл» в цикле — это обычное `if`, а не `throw`).

🤖 Про скорость: исключения **бесплатны, пока не брошены** (помнишь zero-cost из файла про move?). Платишь только в момент ошибки. Поэтому в обычном коде они ничего не замедляют. (В играх/микроконтроллерах их иногда всё же избегают ради предсказуемости — там в ходу коды ошибок и `std::optional`.)

✅ Короткие правила:
- кидай **по значению**, лови **по ссылке** (`catch (const std::exception& e)`);
- **не давай** исключению вылететь из **деструктора** (правило из RAII-файла);
- лови то, что вправду можешь обработать; остальное пусть летит выше.

# 🕵️ Исключения, реверс и безопасность

- Текст ошибки `e.what()` попадает в программу как **строка** (в `.rodata`) — её видно через `strings` (Урок 2.3 в «Под капотом»). ⚠️ Поэтому **не пиши в сообщениях ошибок секретов** (пароли, внутренние пути, ключи) — это утечка.
- Компилятор строит для `try/catch` служебные таблицы — реверсер по ним узнаёт, где в программе обработка ошибок.

## 📋 Шпаргалка

| Слово | Что делает |
|---|---|
| `throw X` | кинуть ошибку `X` (нажать «аварийную кнопку») |
| `try { ... }` | попробовать выполнить блок |
| `catch (const T& e)` | поймать ошибку типа `T` (или его потомка) |
| `e.what()` | текст ошибки |
| `catch (...)` | поймать вообще что угодно |

## 📖 Словарик

| Слово | Простыми словами |
|---|---|
| **Исключение** | Объект-сигнал об ошибке, который летит вверх до `catch` |
| **`throw`** | Кинуть исключение |
| **`try`** | «Попробуй этот блок» |
| **`catch`** | «Поймай ошибку, если будет» |
| **`std::exception`** | Базовый класс всех стандартных ошибок; `catch` по нему ловит все |
| **`logic_error` / `runtime_error`** | Две ветки: баг программиста / внешняя беда во время работы |
| **`invalid_argument`, `out_of_range`…** | Готовые типы из `<stdexcept>` — кидаешь нужный по ситуации |
| **`.what()`** | Метод исключения, возвращает текст ошибки |
| **Код ошибки** | Старый способ: вернуть `-1`/`false` вместо броска |
| **Свой тип исключения** | Класс-наследник `std::runtime_error` с понятным именем и данными |
| **`throw;` (rethrow)** | Пере-бросить пойманное исключение дальше (сохраняя тип) |
| **Срезка (slicing)** | Потеря настоящего типа при `throw e;` вместо `throw;` |

---

> 🤖 Главное: функция не может — пусть **`throw`** кинет ошибку; оборачивай рискованное в **`try`** и обрабатывай в **`catch`**; за **уборку** при этом отвечает **RAII** (само). Исключения — для настоящих ошибок, не для обычной логики.
>
> Будем **дорабатывать**: `noexcept` (обещание «не бросаю»), и современные альтернативы исключениям — `std::optional` / `std::expected`.
