# Часть 1. Проект через `.pro` файл (qmake) ⚙️

## Урок 1.1. Что такое `.pro` и qmake

🧠 В каждом Qt-проекте (на qmake) есть файл с расширением **`.pro`** — это **«паспорт проекта»**. В нём ты простыми строчками описываешь: какие модули Qt нужны, какие файлы входят в проект, какие настройки. Программа **qmake** читает `.pro` и превращает его в настоящие команды сборки для компилятора.

🤖 Раньше мы трогали `.pro` мельком (включали `c++17`). Теперь разберём по-настоящему — это и есть «как правильно создавать Qt-проект».

---

## Урок 1.2. Главные строчки `.pro`

🧠 Вот основные «переменные» файла `.pro`. Их немного, и они читаются по-английски почти как слова:

| Строчка | Что делает |
|---|---|
| `QT += core gui widgets` | подключить **модули** Qt (ядро, графика, виджеты) |
| `QT -= gui` | наоборот, **убрать** модуль (для консоли графика не нужна) |
| `CONFIG += c++17 console` | настройки: стандарт языка, режим консоли |
| `TEMPLATE = app` | что собираем: `app` — программа, `lib` — библиотека |
| `TARGET = MyTool` | имя итогового файла (`MyTool.exe`) |
| `SOURCES += main.cpp` | файлы с кодом (`.cpp`) |
| `HEADERS += window.h` | заголовочные файлы (`.h`) |
| `FORMS += window.ui` | формы окон, нарисованные в Qt Designer |
| `RESOURCES += res.qrc` | картинки, иконки, встроенные файлы |
| `LIBS += -lws2_32` | подключить внешнюю библиотеку |
| `DEFINES += MY_FLAG` | задать макрос (как `#define`) |

✅ **Правило Боба:** добавил в проект **новый класс из Qt** (например сеть) — почти всегда нужно дописать соответствующий **модуль** в строку `QT +=`. Забыл — линкер выдаст «undefined reference». Это подсказка «ты не подключил модуль».

🔬 **Хочешь совсем подробный разбор** каждой строчки `.pro` (что значит `+=`, `-=`, `=`, что делает каждый модуль и настройка) — смотри **«Часть 1.5. Разбор `.pro` под лупой»** в файле `Курс_C++_Проект_Сканер.md`.

---

## Урок 1.3. Минимальный `.pro` для консольного приложения

```pro
QT += core
QT -= gui            # графика не нужна

CONFIG += console c++17
CONFIG -= app_bundle # (для macOS: не делать .app-пакет)

TEMPLATE = app
TARGET = ConsoleTool

SOURCES += main.cpp
```

🤖 Читается так: «нужно только ядро Qt, без графики; собираем консольную программу на C++17; код — в `main.cpp`».

---

## Урок 1.4. Минимальный `.pro` для оконного приложения

```pro
QT += core gui widgets   # ядро + графика + виджеты (окна, кнопки)

CONFIG += c++17

TEMPLATE = app
TARGET = WindowTool

SOURCES += main.cpp
```

⚠️ Разница с консолью всего в одной строке `QT +=`: для окон нужны модули `gui` и `widgets`, и нет `CONFIG += console`.

---

## Урок 1.5. 🛠 Как создать Qt-проект в Qt Creator

1. **File → New Project…**
2. Слева — **Application (Qt)**.
3. Выбери тип:
   - **Qt Console Application** — консольная программа (создаст `.pro` как в Уроке 1.3).
   - **Qt Widgets Application** — оконная программа (создаст `.pro` как в 1.4 + файлы окна).
4. **Build system: qmake** ⚠️ (важно — чтобы получить именно `.pro`-файл; не CMake).
5. Имя, папка → **Next** → выбери **Kit** (Desktop MinGW) → **Finish**.

🤖 Qt Creator создаст проект и `.pro` за тебя. Но теперь, прочитав уроки выше, ты **понимаешь**, что внутри этого файла, и можешь править его руками.

> 📌 Заметка: в новых версиях Qt по умолчанию часто предлагают **CMake** вместо qmake. CMake мощнее, но `.pro` проще читать новичку — поэтому в нашем курсе выбираем **qmake**. Про CMake расскажем, когда будем дорабатывать.

🎯 **Попробуй сам:** создай Qt Console Application и открой его `.pro`. Найди в нём строчки `QT`, `CONFIG`, `SOURCES`. Сравни с Уроком 1.3.

# Часть 2. Консольное Qt-приложение 🖥️

## Урок 2.1. Первый консольный Qt

🧠 В обычном C++ мы печатали через `std::cout`. В Qt есть свои удобные инструменты: `qDebug()` для вывода и `QString` для текста.

```cpp
#include <QCoreApplication>
#include <QDebug>

int main(int argc, char *argv[])
{
    QCoreApplication app(argc, argv);   // «приложение» Qt без графики

    QString imya = "Bob";
    qDebug() << "Privet iz Qt!" << imya;

    return 0;
}
```

🤖 Разбор:
- `QCoreApplication app(...)` — «движок» консольного Qt-приложения. Он нужен для работы таймеров, сети и сигналов.
- `QString` — это «строка по-Qt», умнее обычной `std::string` (умеет Unicode, легко склеивается, конвертируется).
- `qDebug() << ...` — вывод в консоль (удобнее `cout`: сам ставит пробелы и переносы).

---

## Урок 2.2. Цикл событий: когда нужен `app.exec()`

🧠 Важный момент, на котором спотыкаются все новички.

- Если программа просто **что-то напечатала и закончилась** — пиши `return 0;` (как выше).
- Если программа должна **жить и реагировать** (ждать данные из сети, срабатывать по таймеру, обрабатывать сигналы) — нужен **цикл событий**: `return app.exec();`

```cpp
return app.exec();   // программа «крутится» и ждёт события, пока её не закроют
```

🤖 `exec()` — это бесконечный цикл «жди событие → обработай → повтори». Без него сетевые ответы и таймеры просто не успеют сработать. ⚠️ Но если поставить `exec()` в простой «печатающей» программе — она зависнет, ожидая событий, которых нет.

🕵️ **Связь с «Под капотом»:** этот «цикл событий» — то же самое, что главный цикл любой оконной программы Windows (там его крутит WinAPI). Реверсер узнаёт такие циклы в коде по характерным вызовам.

# Часть 3. Оконное (GUI) приложение 🪟

## Урок 3.1. Первое окно с кнопкой

🧠 Теперь самое зрелищное — настоящее окно. Базовые «кирпичики» (виджеты):
- `QApplication` — движок графического приложения (вместо `QCoreApplication`).
- `QWidget` — пустое окно/панель.
- `QLabel` — надпись, `QPushButton` — кнопка, `QLineEdit` — поле ввода.
- `QVBoxLayout` — «раскладка»: расставляет виджеты сверху вниз.

```cpp
#include <QApplication>
#include <QWidget>
#include <QLabel>
#include <QPushButton>
#include <QVBoxLayout>

int main(int argc, char *argv[])
{
    QApplication app(argc, argv);

    QWidget window;
    window.setWindowTitle("Instrument Boba");

    QLabel *label = new QLabel("Nazhmi knopku");
    QPushButton *button = new QPushButton("Nazhmi menya");

    QVBoxLayout *layout = new QVBoxLayout(&window);  // раскладка внутри окна
    layout->addWidget(label);
    layout->addWidget(button);

    // когда кнопку нажмут — поменять надпись (это уже сигнал/слот, Часть 4):
    QObject::connect(button, &QPushButton::clicked, [&]() {
        label->setText("Knopka nazhata!");
    });

    window.show();          // показать окно
    return app.exec();      // цикл событий — окно живёт, пока его не закроют
}
```

🤖 Запусти — появится настоящее окно с кнопкой. Нажми — надпись изменится. Поздравляю, это уже **графическая программа**!

⚠️ Заметь: тут `return app.exec();` **обязателен** — без него окно мелькнёт и сразу закроется (цикл событий не запущен).

> 📌 Окна можно не только писать кодом, но и **рисовать мышкой** в **Qt Designer** (файлы `.ui`). Это удобно для больших окон. Разберём, когда будем дорабатывать курс.

🎯 **Попробуй сам:** добавь второе поле — `QLineEdit` (ввод текста). По нажатию кнопки выведи в надпись то, что пользователь ввёл (`lineEdit->text()`).

## Урок 3.2. Стек или куча в Qt: правило хорошего тона 🏗️

🧠 В Qt есть **своё** правило, где что создавать, и оно немного отличается от обычного C++ (где мы говорили «не пиши `new` руками»). Причина — в Qt у объектов есть **родитель и потомок**.

### Родитель сам убирает за детьми

🧠 Любой `QObject` (а виджеты — это QObject) можно создать **с родителем**. Тогда **родитель становится владельцем**: когда он уничтожается, то **сам удаляет всех своих детей** (и детей детей). Это собственный «RAII» от Qt — целое **дерево объектов**.

🤖 Посмотри на пример из Урока 3.1: окно `window` — **на стеке**, а кнопку, надпись и раскладку мы создали через **`new`**. Раскладка получила родителя (`new QVBoxLayout(&window)`), а `addWidget(...)` сделал окно родителем и для кнопки с надписью. Поэтому, когда `window` уничтожится в конце `main`, оно **само удалит** их всех. Нам **не нужно** писать `delete` — и утечки нет.

### Само правило 📋

| Что | Где создавать | Почему |
|---|---|---|
| **Главное окно и `QApplication`** | на **стеке** в `main()` | живут всю программу, родителя нет |
| **Дочерние виджеты** (кнопки, надписи, поля, раскладки) | в **куче** через `new` + **дать родителя** | должны пережить функцию; родитель удалит сам |
| **Значения** (`QString`, `QColor`, `QPoint`, числа, `QList`…) | на **стеке**, как обычные переменные | это копируемые значения, не часть дерева объектов |

Как в стандартном шаблоне Qt Widgets:
```cpp
int main(int argc, char *argv[]) {
    QApplication app(argc, argv);   // стек
    MainWindow w;                   // стек — главное окно
    w.show();
    return app.exec();
}
```
А дочерние виджеты — в куче, с родителем:
```cpp
auto *knopka = new QPushButton("OK", this);   // куча + родитель this (окно)
auto *layout = new QVBoxLayout(this);          // куча + родитель
layout->addWidget(knopka);
// delete писать НЕ нужно — окно удалит всё само
```

### ⚠️ Три «нельзя»

- ⚠️ **Не давай родителя объекту на стеке.** Тогда и стек, и родитель попытаются его удалить → **двойное удаление** → крах. Виджет с родителем — **всегда в куче**.
- ⚠️ **Не пиши `delete`** для виджета с родителем — родитель сделает это сам (двойной `delete` = крах).
- ⚠️ **Не оборачивай** виджет-с-родителем в `unique_ptr` — он уже под управлением родителя; получится два хозяина.

### А как же «не пиши `new` руками»?

🤖 В обычном C++ мы не любим `new`/`delete` и берём умные указатели (см. [`Курс_C++_Стек_и_куча.md`](Курс_C++_Стек_и_куча.md)). В Qt — **законное исключение**: роль RAII играет **сам Qt** через родителя-потомка. Поэтому `new QPushButton(..., this)` без `delete` — это **правильно и хороший тон**. Умные указатели (или `QScopedPointer`) приберегают для **QObject без родителя** — например, для верхнего окна, которым владеешь ты сам.

✅ **Правило Боба:** **значения — на стек; виджеты — `new` с родителем (Qt удалит сам); главное окно — на стек в `main`.** Никаких ручных `delete` для виджетов с родителем.

🎯 **Попробуй сам:** в окне из Урока 3.1 добавь ещё один `new QLabel("...")` и через `addWidget` положи в раскладку. Запусти, закрой окно — программа завершится чисто: `delete` ты не писал, а утечки нет (окно убрало детей само).

# Часть 4. Сигналы и слоты 🔌

## Урок 4.1. Главная идея Qt

🧠 **Сигналы и слоты** — это сердце Qt, способ объектам «разговаривать» друг с другом.

- **Сигнал** (*signal*) — объект кричит: «**со мной кое-что случилось!**» (например, кнопку нажали → сигнал `clicked`).
- **Слот** (*slot*) — функция, которая **реагирует** на сигнал.
- **`connect`** — «провод», соединяющий сигнал с реакцией.

🤖 Аналогия: сигнал — это **дверной звонок** (событие «кто-то пришёл»), слот — твоё действие «пойти открыть дверь», а `connect` — провод между кнопкой звонка и звоночком. Нажали кнопку → провод сработал → звонок зазвонил.

---

## Урок 4.2. Современный синтаксис `connect`

```cpp
connect(ОТПРАВИТЕЛЬ, &Класс::сигнал,  ПОЛУЧАТЕЛЬ, &Класс::слот);
```

Пример из окна выше:
```cpp
QObject::connect(button, &QPushButton::clicked,   // сигнал «кнопку нажали»
                 label, &QLabel::clear);           // слот «очистить надпись»
```
Или с лямбдой (мини-функцией прямо на месте — помнишь из углублённого файла?):
```cpp
QObject::connect(button, &QPushButton::clicked, [&]() {
    qDebug() << "Knopku nazhali!";
});
```

✅ **Правило Боба:** используй **новый** синтаксис со стрелочками `&Класс::имя` (function-pointer). Старый со строками (`SIGNAL(...)/SLOT(...)`) тоже встречается в старом коде, но в нём ошибки видны только при запуске, а в новом — ловит компилятор. Правильность!

---

## Урок 4.3. Свой сигнал (коротко про `Q_OBJECT` и moc)

🧠 Можно делать и **свои** сигналы. Для этого класс наследуют от `QObject` и пишут магическое слово `Q_OBJECT`:

```cpp
// schetchik.h
#include <QObject>

class Schetchik : public QObject
{
    Q_OBJECT                    // ← обязательный «магический» макрос
public:
    void uvelichit() {
        ++znachenie_;
        emit izmenilos(znachenie_);   // «испустить» сигнал
    }

signals:                        // тут только ОБЪЯВЛЯЮТ сигналы (без тела!)
    void izmenilos(int novoe);

private:
    int znachenie_ = 0;
};
```

🤖 Что происходит:
- `signals:` — раздел, где перечислены сигналы. Тело им писать не надо — Qt сгенерирует его сам.
- `emit izmenilos(...)` — «послать сигнал» всем, кто на него подписан через `connect`.
- **`Q_OBJECT`** включает магию. А делает эту магию тот самый **moc** (Meta-Object Compiler): перед сборкой он читает твой `.h`, видит `Q_OBJECT`/`signals` и **дописывает за тебя** дополнительный C++-код для сигналов.

⚠️ Класс с `Q_OBJECT` правильно держать в **отдельных файлах** `.h`/`.cpp` (не в `main.cpp`) — тогда moc подхватит его автоматически.

🕵️ **Реверс-смысл:** moc встраивает в программу **имена сигналов и слотов как строки** (мета-информацию). Поэтому Qt-программы легко узнать при реверсе: в них видны строки вроде `clicked()`, имена классов `Q...`, и импорты библиотек `Qt6Core.dll`/`Qt6Widgets.dll`. По этим следам аналитик сразу понимает: «это написано на Qt».

# Часть 5. Обзор модулей Qt (вкратце) 🧰

🧠 Qt огромен и поделён на **модули**. Подключаешь нужный — строкой `QT += имя` в `.pro`. Вот основные:

**🧱 Фундамент (нужны почти всегда):**

| Модуль (`QT +=`) | Для чего | Ключевые классы |
|---|---|---|
| **core** | ядро: строки, контейнеры, файлы, сигналы/слоты, таймеры, JSON, настройки | `QString`, `QList`/`QMap`, `QFile`, `QObject`, `QTimer`, `QJsonDocument`, `QSettings` |
| **gui** | низкоуровневая графика: картинки, рисование, шрифты, цвета | `QImage`, `QPainter`, `QColor`, `QFont` |
| **widgets** | классические окна и элементы интерфейса | `QMainWindow`, `QPushButton`, `QLabel`, `QLineEdit`, `QTableView` |

**🎨 Современный («текучий») интерфейс:**

| Модуль (`QT +=`) | Для чего | Ключевые классы |
|---|---|---|
| **qml** | язык QML + движок JavaScript для интерфейса | `QQmlApplicationEngine` |
| **quick** | анимированный «тач»-интерфейс (моб./планшеты) на QML | `QQuickView`, QML-компоненты |

**🌐 Сеть, данные, устройства:**

| Модуль (`QT +=`) | Для чего | Ключевые классы |
|---|---|---|
| **network** | сеть: TCP/UDP, HTTP-запросы | `QTcpSocket`, `QTcpServer`, `QNetworkAccessManager`, `QHostInfo` |
| **sql** | базы данных (встроен SQLite) | `QSqlDatabase`, `QSqlQuery`, `QSqlTableModel` |
| **websockets** | двусторонняя связь с веб-сервером (WebSocket) | `QWebSocket` |
| **serialport** 🔧 | связь с «железом» по COM-порту (Arduino, датчики) | `QSerialPort`, `QSerialPortInfo` |

**⚙️ Параллельность и мультимедиа:**

| Модуль (`QT +=`) | Для чего | Ключевые классы |
|---|---|---|
| **concurrent** | фоновые задачи / многопоточность по-простому | `QtConcurrent::run`, `QFuture` |
| **multimedia** | звук, видео, камера | `QMediaPlayer`, `QCamera`, `QAudioOutput` |

**📦 Прочее полезное:**

| Модуль (`QT +=`) | Для чего | Ключевые классы |
|---|---|---|
| **charts** | графики и диаграммы | `QChart`, `QLineSeries` |
| **svg** | показ векторных картинок SVG | `QSvgRenderer`, `QSvgWidget` |
| **printsupport** | печать на принтер | `QPrinter`, `QPrintDialog` |
| **testlib** | автотесты (проверять, что код работает) ✅ | `QTest` |
| **webenginewidgets** | встроить браузер Chromium, показывать веб-страницы | `QWebEngineView` |

🤖 **Как узнать, какой модуль нужен классу?** На странице любого класса в документации Qt сверху написана строчка `QT += ...` — это и есть нужный модуль (помнишь Правило: добавил класс → допиши модуль в `.pro`, иначе линкер ругнётся «undefined reference»).

🤖 Тебе для старта почти всегда хватит трёх: **core** (везде), **widgets** (для окон), **network** + **sql** (для наших «хакерских» тем — сеть и базы). Их и разберём дальше чуть подробнее.

# Часть 6. Сеть в Qt (с нашим уклоном) 🌐

## Урок 6.1. Помнишь боль Winsock? Qt проще

🧠 В файле «Под капотом» (Часть 4) мы подключались к серверу через Winsock — там было `WSAStartup`, ручной сокет, `WSACleanup`… длинно и легко ошибиться. В Qt то же самое — несколько строк:

```cpp
// .pro:  QT += network
#include <QTcpSocket>
#include <QDebug>

QTcpSocket socket;
socket.connectToHost("example.com", 80);     // подключиться (хост, порт)

if (socket.waitForConnected(3000)) {          // ждать до 3 секунд
    qDebug() << "Podklyuchilis!";
} else {
    qDebug() << "Ne udalos:" << socket.errorString();
}
```

🤖 Никаких `WSAStartup`/`WSACleanup` — Qt берёт всю возню на себя (это RAII-стиль: объект `socket` сам приберёт за собой). Вот зачем нужны фреймворки: они **прячут сложность** ОС за удобными классами.

🕵️ **Реверс-смысл:** но «под капотом» Qt всё равно вызывает те же `socket`/`connect`/`send` из `ws2_32.dll`! Поэтому реверсер увидит в импортах и Winsock-функции, и Qt-сетевые классы. Удобство для программиста, но «отпечатки» в файле остаются те же.

---

## Урок 6.2. HTTP-запрос (скачать страницу)

🧠 Для веба (HTTP) в Qt есть `QNetworkAccessManager`. Он работает **асинхронно** — через сигнал «ответ пришёл» (вот зачем нужны сигналы/слоты и `exec()`!).

```cpp
// .pro:  QT += network
#include <QNetworkAccessManager>
#include <QNetworkReply>

QNetworkAccessManager manager;
QObject::connect(&manager, &QNetworkAccessManager::finished,
                 [](QNetworkReply *reply) {
    qDebug() << "Otvet poluchen, bayt:" << reply->readAll().size();
    reply->deleteLater();
});
manager.get(QNetworkRequest(QUrl("http://example.com")));
```

⚠️ Применяй такие инструменты только к **своим** серверам или к публичным сайтам, которые разрешают это (например тренировочным). Заваливать чужой сайт запросами — это атака, и это незаконно.

# Часть 7. Базы данных в Qt 🗄️

## Урок 7.1. Зачем база данных

🧠 Переменные живут, пока программа запущена. Чтобы **сохранить данные надолго** (список пользователей, логи, настройки), их пишут на диск. Самый простой способ — **обычный файл** (Часть 10). А когда данных много и нужно удобно искать/фильтровать — берут **базу данных**. Самая простая — **SQLite**: целая база в одном файле, и она **уже встроена** в Qt.

```cpp
// .pro:  QT += sql
#include <QSqlDatabase>
#include <QSqlQuery>
#include <QDebug>

QSqlDatabase db = QSqlDatabase::addDatabase("QSQLITE");
db.setDatabaseName("data.db");        // база — это один файл data.db
db.open();

QSqlQuery q;
q.exec("CREATE TABLE IF NOT EXISTS users (id INTEGER PRIMARY KEY, name TEXT)");
```

🤖 Команды для базы пишут на специальном языке **SQL** (`CREATE TABLE`, `INSERT`, `SELECT`…). Это отдельный маленький язык — основы дадим, когда будем дорабатывать. Сейчас — самое важное про **безопасность**.

---

## Урок 7.2. 🔓 SQL-инъекция — главная уязвимость веба

🧠 Это, пожалуй, **самый знаменитый способ взлома сайтов**. Понять его — обязательно для белого хакера.

❌ **КАК НЕЛЬЗЯ** — склеивать запрос из текста пользователя:
```cpp
QString imya = poleVvoda->text();           // пользователь ввёл это
q.exec("SELECT * FROM users WHERE name = '" + imya + "'");   // 💥 ОПАСНО!
```
Что будет, если пользователь введёт хитрую строку вроде `' OR '1'='1`? Запрос превратится в `... WHERE name = '' OR '1'='1'` — а это **правда всегда**, и база отдаст **всех** пользователей. Так воруют целые базы паролей. Это и есть **SQL-инъекция**.

✅ **КАК ПРАВИЛЬНО** — подготовленный запрос с «заглушками» `?`:
```cpp
q.prepare("SELECT * FROM users WHERE name = ?");
q.addBindValue(imya);                        // Qt сам всё безопасно подставит
q.exec();
```
Теперь, что бы пользователь ни ввёл, это считается **просто текстом**, а не частью команды. Инъекция невозможна.

✅ **Правило Боба (золотое для безопасности):** **никогда** не склеивай SQL-запрос из пользовательского ввода строками. **Всегда** используй подготовленные запросы (`prepare` + `addBindValue`). Это одно правило закрывает целый класс взломов.

🕵️ Понимать инъекцию надо, чтобы **находить и чинить** такие дыры (с разрешения!) и писать безопасный код. Это классическая задача на CTF и в работе пентестера.

# Часть 8. SerialPort — общаемся с «железом» (Arduino) 🔌🔧

🧠 Модуль **`serialport`** позволяет программе разговаривать с устройствами по **COM-порту** (последовательный порт): **Arduino**, датчики, микроконтроллеры, GPS-модули. Для любителя техники — самое вкусное: можно из C++ зажечь светодиод на Arduino.

🔧 Аналогия: COM-порт — это «провод-телефон» между компьютером и устройством: по нему шлёшь байты туда и принимаешь обратно.

🛠 **`.pro`:** `QT += serialport`. Главные классы: `QSerialPortInfo` (узнать порты), `QSerialPort` (само соединение).

## Урок 8.1. Найти, на каком порту устройство

```cpp
#include <QCoreApplication>
#include <QSerialPortInfo>
#include <QDebug>

int main(int argc, char* argv[]) {
    QCoreApplication app(argc, argv);
    for (const QSerialPortInfo& info : QSerialPortInfo::availablePorts()) {
        qDebug() << info.portName() << "-" << info.description();
    }
    return 0;
}
```

🤖 Выведет список портов, например `"COM3" - "Arduino Uno"`. Так ты узнаёшь имя порта своего устройства.

## Урок 8.2. Послать команду и прочитать ответ

```cpp
#include <QSerialPort>
#include <QDebug>

QSerialPort port;
port.setPortName("COM3");        // имя из прошлого примера
port.setBaudRate(9600);          // ⚠️ скорость должна совпадать с прошивкой Arduino!

if (port.open(QIODevice::ReadWrite)) {
    port.write("LED_ON\n");                  // послали команду устройству
    port.waitForBytesWritten(100);

    if (port.waitForReadyRead(1000)) {       // ждём ответ до 1 секунды
        qDebug() << "Arduino otvetil:" << port.readAll();
    }
    port.close();
} else {
    qDebug() << "Ne udalos otkryt port:" << port.errorString();
}
```

🤖 Разбор:
- `setBaudRate(9600)` — «скорость» порта; она **обязана совпадать** с той, что в скетче Arduino (`Serial.begin(9600)`), иначе придёт «каша».
- `write(...)` — отправить байты; `readAll()` — забрать пришедшее.
- ⚠️ `waitForReadyRead` **блокирует** (синхронно) — для окна так нельзя (заморозит интерфейс). Правильно — асинхронно, по **сигналу `readyRead`** (привет файлам про сигналы/слоты и асинхронность).
- 🔧 `port.close()` лучше доверить **RAII** (закрыть в деструкторе / через scope guard — см. RAII-файл), чтобы не забыть.

🎯 **Попробуй сам:** залей в Arduino скетч, который по команде `"LED_ON\n"` зажигает светодиод и отвечает `"OK"`. Из C++ пошли команду и прочитай ответ.

# Часть 9. Multimedia — звук и видео 🔊

🧠 Модуль **`multimedia`** проигрывает звуки, музыку и видео, работает с камерой. Два главных инструмента для звука:
- **`QSoundEffect`** — короткие звуки с мгновенным откликом (пик, сигнал, эффект). Играет `WAV`.
- **`QMediaPlayer`** — музыка и видео из файлов (`mp3`, `mp4`…).

🛠 **`.pro`:** `QT += multimedia`.

## Урок 9.1. «Бип» при событии (`QSoundEffect`)

🔧 Например, пусть сканер **пикает**, когда нашёл открытый порт.

```cpp
#include <QCoreApplication>
#include <QSoundEffect>
#include <QUrl>

int main(int argc, char* argv[]) {
    QCoreApplication app(argc, argv);

    QSoundEffect bip;
    bip.setSource(QUrl::fromLocalFile("beep.wav"));
    bip.setVolume(0.8);
    bip.play();                         // звук играет асинхронно!

    // выйдем, когда звук доиграет:
    QObject::connect(&bip, &QSoundEffect::playingChanged, [&]{
        if (!bip.isPlaying()) app.quit();
    });
    return app.exec();                  // ⚠️ без цикла событий звук не успеет сыграть
}
```

🤖 Важное: звук играет **асинхронно** (в фоне), поэтому нужен **цикл событий** (`app.exec()`) — иначе программа закончится раньше, чем звук сыграет. Это та самая асинхронность из отдельного файла.

## Урок 9.2. Проиграть музыку (`QMediaPlayer`, Qt 6)

```cpp
#include <QMediaPlayer>
#include <QAudioOutput>
#include <QUrl>

QMediaPlayer player;
QAudioOutput audio;                     // в Qt 6 звук идёт через отдельный «выход»
player.setAudioOutput(&audio);
audio.setVolume(0.5);

player.setSource(QUrl::fromLocalFile("music.mp3"));
player.play();                          // тоже нужен цикл событий
```

🤖 В Qt 6 у плеера звук выводится через отдельный объект `QAudioOutput` (в Qt 5 было иначе — частая путаница). `QSoundEffect` — для коротких эффектов, `QMediaPlayer` — для музыки и видео.

⚠️ Нужны сами файлы (`beep.wav`, `music.mp3`) рядом с программой и работающий цикл событий.

🎯 **Попробуй сам:** добавь в свой сканер портов `QSoundEffect`, чтобы он пикал на каждом найденном открытом порту.

# Часть 10. Работа с файлами 📄

🧠 Файл — самый простой способ **сохранить данные**, которые переживут закрытие программы (логи, настройки, результаты сканирования). Это даже проще, чем база данных (Часть 7). В Qt файлы — это класс `QFile`, а удобнее всего читать/писать текст через `QTextStream`. Всё это в модуле **core** — ничего доустанавливать не нужно.

🔧 На нашу тему: сохраним результаты сканера портов в лог-файл.

## Урок 10.1. Записать текст в файл

```cpp
#include <QFile>
#include <QTextStream>

QFile fayl("scan.log");
if (fayl.open(QIODevice::WriteOnly | QIODevice::Text)) {
    QTextStream out(&fayl);
    out << "port 80 OTKRYT\n";
    out << "port 443 OTKRYT\n";
}   // fayl закроется САМ при выходе из { } — это RAII!
```

🤖 Разбор:
- `open(...)` возвращает `true/false` — **всегда проверяй** (вдруг нет прав или папки).
- `WriteOnly` — пишем; `Text` — правильно обрабатывать переносы строк на Windows/Mac.
- `QTextStream out(&fayl)` — удобный «принтер» в файл (как `cout`, только в файл).
- ⚠️ `WriteOnly` **стирает** старое содержимое. Чтобы **дописать** в конец — режим `Append` (Урок 10.3).
- 🔧 Заметь: мы **не вызывали `close()`** — `QFile` закрывает файл сам в деструкторе. Это **RAII** (см. RAII-файл).

## Урок 10.2. Прочитать файл

```cpp
#include <QFile>
#include <QTextStream>
#include <QDebug>

QFile fayl("scan.log");
if (fayl.open(QIODevice::ReadOnly | QIODevice::Text)) {
    QTextStream in(&fayl);
    while (!in.atEnd()) {                 // пока не конец файла
        QString stroka = in.readLine();   // читаем по строчке
        qDebug() << stroka;
    }
}
```

🤖 `readLine()` читает по одной строке. Нужен **весь файл сразу** — `QString ves = in.readAll();`.

## Урок 10.3. Дописать в конец (Append)

```cpp
QFile fayl("scan.log");
if (fayl.open(QIODevice::Append | QIODevice::Text)) {   // добавить в конец, не стирая
    QTextStream out(&fayl);
    out << "port 22 OTKRYT\n";
}
```

🤖 `Append` дописывает в конец, не трогая старое. Идеально для **логов**, которые растут со временем.

## Урок 10.4. Проверить файл и посмотреть папку

```cpp
#include <QFile>
#include <QFileInfo>
#include <QDir>
#include <QDebug>

// есть ли файл и какого он размера:
if (QFile::exists("scan.log")) {
    qDebug() << "Razmer:" << QFileInfo("scan.log").size() << "bayt";
}

// какие файлы лежат в текущей папке:
for (const QString& imya : QDir(".").entryList(QDir::Files)) {
    qDebug() << imya;
}
```

🤖 `QFileInfo` — справка о файле (существует ли, размер, имя). `QDir` — работа с папками (список файлов, создать/удалить папку).

> 📌 На будущее: для **надёжной** записи (чтобы файл не испортился, если программа упадёт посреди записи) есть `QSaveFile` — он пишет «начисто» и подменяет файл целиком. Разберём, когда будем дорабатывать.

🎯 **Попробуй сам:** допиши свой сканер так, чтобы он **сохранял** найденные открытые порты в `scan.log` (режим `Append`), а отдельной командой **читал** и показывал его содержимое.
