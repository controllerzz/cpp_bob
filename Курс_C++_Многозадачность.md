# Часть 1. Что такое многозадачность 🔀

## Урок 1.1. Делать несколько дел «сразу»

🧠 Есть два похожих, но **разных** слова:

- **Конкурентность** (concurrency) — *жонглирование* задачами: один работник очень быстро переключается между делами. Кажется, что всё идёт сразу, но в каждый момент он делает что-то одно.
- **Параллелизм** (parallelism) — *по-настоящему одновременно*: несколько работников, каждый за своим делом.

🏭 Технический образ:
- **Один рабочий, много станков**: он бегает между станками и переключается. Это **конкурентность** (так работает одно ядро процессора).
- **Много рабочих, у каждого свой станок**: реально одновременно. Это **параллелизм** (так работают несколько ядер).

🤖 У современного процессора несколько **ядер** — значит, он умеет по-настоящему параллельно. Многозадачность — это умение программы загрузить все ядра и/или не простаивать, пока чего-то ждёт.

---

## Урок 1.2. Процесс и поток 🏭🧵

🧠 Два главных слова всей темы:

- **Процесс** — это **целая запущенная программа** со **своей собственной памятью**. У каждого процесса свой код, свой стек, своя куча (помнишь карту памяти из «Под капотом»?). Процессы друг друга **не видят** напрямую.
- **Поток** — это **работник внутри одного процесса**. В одном процессе может быть много потоков, и они **делят общую память** этого процесса.

🏭 Технический образ:
- **Процесс = отдельный цех** со своим складом (память). Чужой склад не виден.
- **Поток = рабочий внутри цеха**. Несколько рабочих в одном цехе пользуются **общим складом**.

🧠 Очень важная деталь (связь с «Под капотом»): потоки **делят** код, статическую память и кучу, **но у каждого потока свой собственный стек**.

```
   ПРОЦЕСС (один цех)
   ┌───────────────────────────────────────┐
   │  общий склад: КОД · СТАТИКА · КУЧА    │  ← делят все потоки
   ├───────────────┬───────────┬───────────┤
   │ стек потока 1 │ стек пот.2│ стек пот.3│  ← у каждого свой
   └───────────────┴───────────┴───────────┘
```

🤖 Отсюда сразу два вывода:
- потокам **легко** общаться (общий склад — общие переменные);
- но это же **опасно**: два потока могут полезть к одной «полке» одновременно и устроить путаницу. Об этом — Часть 3.

---

## Урок 1.3. Зачем многозадачность нужна

🧠 Три главные причины:

1. **Ускорить** тяжёлую работу — раскидать её по ядрам (параллелизм).
2. **Не зависать** — пока программа делает что-то долгое, окно не должно «замерзать». Долгую работу убирают в отдельный поток (привет Qt-окнам!).
3. **Не простаивать на ожидании** — пока ждём ответа из сети или с диска, можно заниматься другими делами. (Помнишь, сканер портов был медленным, потому что ждал каждый порт по очереди? Это лечится многозадачностью!)

# Часть 2. Потоки — несколько работников в программе 🧵

## Урок 2.1. Запускаем поток

🧠 В C++ поток — это `std::thread`. Даёшь ему функцию — он начинает её выполнять **параллельно** с `main`.

```cpp
#include <iostream>
#include <thread>

void rabota(int nomer) {
    std::cout << "Potok " << nomer << " rabotaet\n";
}

int main() {
    std::thread t1(rabota, 1);   // запустили работника 1
    std::thread t2(rabota, 2);   // запустили работника 2

    t1.join();                   // дождаться, пока работник 1 закончит
    t2.join();                   // дождаться работника 2

    std::cout << "Vse gotovo\n";
}
```

🤖 `join()` — «подожди этот поток». Без него `main` мог бы закончиться раньше работников, и программа упала бы.

⚠️ Запусти несколько раз — увидишь, что строчки иногда **перемешиваются** (`Potok 2` раньше `Potok 1`). Это нормально: потоки идут одновременно, и порядок не гарантирован. А ещё это **первый намёк на проблему** из Части 3.

🛠 **Про `.pro`:** на Windows (MinGW) обычно ничего дополнительно не нужно, только `CONFIG += c++17`. На Linux добавь `QMAKE_CXXFLAGS += -pthread` и `LIBS += -pthread`.

✅ **Правило Боба:** поток — для того, что можно делать **независимо** и **параллельно**: ускорить вычисления, делать фоновую работу, ждать сеть, не блокируя остальное.

🎯 **Попробуй сам:** вспомни сканер портов. Если каждый порт проверять в **отдельном** потоке, сканирование станет в разы быстрее. Подумай, какие части сканера можно делать параллельно.

# Часть 3. Проблема: гонки за общими данными 🏁

## Урок 3.1. Когда двое лезут к одной полке

🧠 Потоки делят общую память. Что будет, если два потока меняют **одну и ту же** переменную одновременно?

```cpp
#include <iostream>
#include <thread>

int schyot = 0;          // ОБЩАЯ переменная

void dobavit() {
    for (int i = 0; i < 100000; ++i) {
        ++schyot;        // ⚠️ оба потока трогают одно и то же!
    }
}

int main() {
    std::thread t1(dobavit);
    std::thread t2(dobavit);
    t1.join();
    t2.join();
    std::cout << "Ozhidali 200000, poluchili: " << schyot << "\n";
    // часто получится МЕНЬШЕ 200000!
}
```

🤖 Почему не 200000? Потому что `++schyot` — это на самом деле **три действия**: «прочитай число → прибавь 1 → запиши обратно». Если два потока делают это вперемешку, они затирают работу друг друга.

🏭 Технический образ: два рабочих записывают новое число в **один журнал**. Оба прочитали «5», оба написали «6» — а должно было стать «7». Один результат потерялся. Это называется **гонка** (*race condition*).

🕵️ **Безопасность:** гонки — это целый **класс уязвимостей** (например *TOCTOU* — «проверил одно, а к моменту использования стало другое»). Белые хакеры специально их ищут. Так что понимать гонки нужно и для надёжности, и для защиты.

# Часть 4. Мьютекс — «один за раз» 🔒

## Урок 4.1. Что такое мьютекс

🧠 **Мьютекс** (*mutex* = *mutual exclusion*, «взаимное исключение») — это «жезл», который гарантирует: к общим данным в каждый момент прикасается **только один** поток. Остальные ждут своей очереди.

🏭 Технический образ: на стройке один **жезл** (или единственный ключ от пультовой). У кого жезл — тот и работает с общим журналом; остальные стоят и ждут, пока жезл освободится.

```cpp
#include <iostream>
#include <thread>
#include <mutex>

int schyot = 0;
std::mutex m;            // наш «жезл»

void dobavit() {
    for (int i = 0; i < 100000; ++i) {
        std::lock_guard<std::mutex> zamok(m);  // взять жезл
        ++schyot;                              // теперь трогаем безопасно
    }                                          // выйдя из { } — жезл сам вернётся
}

int main() {
    std::thread t1(dobavit);
    std::thread t2(dobavit);
    t1.join();
    t2.join();
    std::cout << "Poluchili: " << schyot << "\n";   // теперь ВСЕГДА 200000
}
```

🤖 `std::lock_guard` — это **RAII** (помнишь Часть D углублённого файла?): он берёт жезл при создании и **сам отпускает** его, когда выходит из фигурных скобок. Забыть «вернуть жезл» невозможно — как с умным указателем.

## Урок 4.2. Цена и опасности

🧠 Мьютекс защищает данные, но у него есть цена:
- пока один держит жезл — **остальные ждут**, то есть теряется часть скорости (поэтому держи блокировку **как можно короче**);
- **взаимная блокировка** (*deadlock*): два потока схватили по одному жезлу и ждут второй — у каждого тот, что нужен другому. Оба стоят **вечно**. 🏭 Как два рабочих: один взял дрель и ждёт шуруповёрт, другой взял шуруповёрт и ждёт дрель.

✅ **Правило Боба:** общие данные трогает **только один за раз** — через мьютекс. Держи блокировку коротко. А ещё лучше — по возможности **вообще не делить данные** (см. Часть 5: очереди).

# Часть 5. Очереди — безопасный конвейер между потоками 📦➡️

## Урок 5.1. Производитель и потребитель

🧠 Часто работу удобно организовать так: один поток **создаёт** задания, другой их **выполняет**. Связать их безопаснее всего через **очередь** (FIFO — помнишь из углублённого файла?).

🏭 Технический образ: **конвейерная лента** между двумя рабочими. Первый кладёт детали на ленту, второй снимает и обрабатывает. Они не толкаются у одного стола — лента их **развязывает**.

```cpp
#include <queue>
#include <mutex>

std::queue<int> zadachi;     // общая очередь заданий
std::mutex m;

// ПРОИЗВОДИТЕЛЬ (один поток) кладёт задание:
void polozhit(int z) {
    std::lock_guard<std::mutex> zamok(m);
    zadachi.push(z);
}

// ПОТРЕБИТЕЛЬ (другой поток) забирает задание:
bool vzyat(int& kuda) {
    std::lock_guard<std::mutex> zamok(m);
    if (zadachi.empty()) return false;
    kuda = zadachi.front();
    zadachi.pop();
    return true;
}
```

🤖 Очередь по-прежнему общая, поэтому её защищает мьютекс. Но теперь потоки общаются **аккуратно**: один только кладёт, другой только берёт. Никто не лезет в чужие переменные.

> 📌 В «настоящем» коде добавляют `std::condition_variable`, чтобы потребитель **спал**, пока заданий нет, а не дёргал очередь впустую. Это чуть сложнее — разберём, когда будем дорабатывать.

✅ **Правило Боба:** вместо «все лезут в общие переменные с кучей мьютексов» — **передавай работу через очередь**. Чище, безопаснее и понятнее.

# Часть 6. Процессы и межпроцессное взаимодействие (IPC) 📨

## Урок 6.1. У процессов память не общая

🧠 Потоки делят память — а **процессы нет**. У каждого процесса свой склад (Урок 1.2). Поэтому просто «поделить переменную» между двумя программами **нельзя** — нужны специальные каналы связи. Это и есть **IPC** (*Inter-Process Communication*, межпроцессное взаимодействие).

## Урок 6.2. Способы IPC

| Способ | Что это | Технический образ |
|---|---|---|
| **Файл** | один пишет файл, другой читает | оставить записку на столе |
| **Канал (pipe)** | «труба» между программами (в консоли это `|`) | труба между цехами |
| **Сокеты** | связь даже между **разными компьютерами** | телефон / рация |
| **Разделяемая память** | общий кусок памяти на два процесса (+ нужен мьютекс/семафор) | общая доска на стене между цехами |
| **Именованный канал / локальный сокет** | «труба» с именем внутри одного компьютера | внутренний телефон |

🤖 Заметь: **сокеты** мы уже знаем (сканер, Qt-сеть)! Сетевое общение — это тоже IPC, просто процессы могут быть на разных машинах.

## Урок 6.3. 🛠 Самый простой IPC: запустить программу и прочитать её вывод

🧠 В Qt есть класс `QProcess` — он **запускает другую программу** и читает её вывод через канал (pipe). Это уже настоящий IPC.

```cpp
// .pro:  QT += core
#include <QProcess>
#include <QDebug>

QProcess p;
p.start("ping", QStringList() << "-n" << "1" << "127.0.0.1");  // -n для Windows
p.waitForFinished();

QString vyvod = p.readAllStandardOutput();   // получили вывод чужой программы!
qDebug() << vyvod;
```

🤖 Наша программа запустила `ping`, дождалась его и забрала результат. Это и есть «два процесса поговорили». (Помнишь, в сканере сети я обещал показать `ping` через `QProcess`? Вот он.)

## Урок 6.4. Зачем процессы, если есть потоки?

🧠 Главное слово — **изоляция**. У процессов раздельная память, и это даёт:

- **Надёжность:** упал один процесс — **остальные живут**. Если поток уронит процесс — падает вся программа целиком.
- **Безопасность (песочница):** опасную работу выносят в отдельный процесс с урезанными правами. 🕵️ Именно поэтому **браузеры** делают каждую вкладку **отдельным процессом** — взломанная вкладка не доберётся до остальных.
- **Масштаб:** процессы можно раскидать по разным компьютерам и связать сокетами.

# Часть 7. Когда что выбирать (зачем где преимущества) 🧭

## Потоки vs процессы

| | Потоки 🧵 | Процессы 🏭 |
|---|---|---|
| Память | **общая** | у каждого **своя** |
| Общаться | легко (общие переменные) | только через IPC |
| Создавать | дёшево и быстро | дороже |
| Если один упал | падает **вся** программа | остальные **живут** |
| Безопасность/изоляция | слабая | **сильная** (песочница) |
| Главный риск | гонки за данными → нужен мьютекс | сложнее наладить связь |

## Шпаргалка «что выбрать»

- Ускорить вычисления на всех ядрах → **потоки** (или процессы).
- Не заморозить окно при долгой работе → долгую работу в **отдельный поток**.
- Ждать сеть/диск, не простаивая → **потоки** (или асинхронность).
- Нужна надёжность и безопасность (изоляция) → **процессы**.
- Связать **разные программы** → **IPC** (pipe / `QProcess`).
- Связь по сети / между машинами → **сокеты**.
- Потоки общаются → лучше **очередь**; общие данные защищай **мьютексом**.

# Часть 8. Qt-способ — короче и безопаснее 🪟

🤖 В Qt многозадачность часто делают **проще и безопаснее**, чем вручную с мьютексами:

- **`QtConcurrent::run`** — запустить функцию в фоновом потоке одной строкой (нужен `QT += concurrent`):
  ```cpp
  #include <QtConcurrent>
  QtConcurrent::run([]() {
      // тяжёлая работа в фоне — окно НЕ зависает
  });
  ```
- **`QThread`** — класс-поток с сигналами/слотами.
- **Сигналы и слоты между потоками** — это «фишка» Qt: фоновый поток через **сигнал** безопасно передаёт результат в главный поток, **часто вообще без ручных мьютексов**. Для окон это идеально: считаем в фоне, а готовый результат сигналом отдаём окну.
- **`QMutex`, `QSemaphore`, `QWaitCondition`** — Qt-версии мьютекса и компании.
- **`QProcess` / `QSharedMemory` / `QLocalSocket`** — IPC по-Qt (запуск программ, общая память, локальные «трубы»).

✅ **Правило Боба:** в Qt для фоновой работы бери **`QtConcurrent` + сигналы/слоты**. Ручные потоки и мьютексы — только когда вправду нужно. Меньше ручных блокировок → меньше багов.

🎯 **Большой проект (на потом):** ускорить наш сканер портов — проверять много портов **параллельно** через `QtConcurrent`, а найденные открытые порты отдавать в окно **сигналом**. Это и есть «взрослая» версия из файла про сканер.

# 🕵️ Многозадачность и реверс

🧠 Несколько связей с реверсом и безопасностью (понимаем, чтобы защищать):

- Многопоточные программы **труднее реверсить**: код идёт параллельно, порядок «плавает», поведение зависит от времени. Иногда это используют, чтобы запутать аналитика.
- Аналитик следит за созданием потоков и процессов — в импортах это `CreateThread`, `CreateProcess`, `CreateRemoteThread` (помнишь триаж по импортам из «Под капотом»? `CreateRemoteThread` в чужом процессе — большой красный флаг 🚩).
- **Гонки (race / TOCTOU)** — реальный класс уязвимостей, который ищут белые хакеры.
- **Изоляция процессов** — это защита (песочница). Понимая её, понимаешь, как устроена безопасность браузеров и ОС.

## 📖 Словарик

| Слово | Простыми словами |
|---|---|
| **Конкурентность** | Жонглирование задачами с переключением (на одном ядре) |
| **Параллелизм** | По-настоящему одновременно (на нескольких ядрах) |
| **Процесс** | Целая программа со своей памятью (отдельный «цех») |
| **Поток (thread)** | Работник внутри процесса; делит память с другими |
| **Гонка (race condition)** | Двое меняют общие данные разом → путаница |
| **Мьютекс** | «Жезл»: к общим данным — только один за раз |
| **`lock_guard`** | RAII-замок: сам берёт и сам отпускает мьютекс |
| **Deadlock** | Взаимная блокировка: оба ждут друг друга вечно |
| **Очередь заданий** | Конвейер между потоками (производитель → потребитель) |
| **IPC** | Связь между процессами (файл, pipe, сокет, общая память) |
| **Песочница** | Изоляция в отдельном процессе ради безопасности |
| **`QtConcurrent`** | Простой способ запустить работу в фоне в Qt |

> 🤖 Это **первая версия** файла про многозадачность — будем **дорабатывать**: добавим `std::async`/`std::future`, `condition_variable` (умная очередь), атомарные переменные (`std::atomic`), пулы потоков, подробный `QThread` с сигналами/слотами и параллельный сканер целиком.
>
> Запомни три вещи: **поток** — для скорости и «не зависать»; **мьютекс/очередь** — чтобы потоки не подрались за данные; **процесс/IPC** — для изоляции, надёжности и безопасности. 🛡️
