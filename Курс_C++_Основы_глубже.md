# Часть A. Правильность с самого начала 🎯

> Эти привычки — фундамент. Если усвоить их сразу, в будущем ты избежишь сотен багов.

## Урок A.1. Всегда инициализируй переменные

🧠 Когда ты пишешь `int x;` и **не** задаёшь значение — в коробочке `x` лежит **мусор**: какое-то случайное число, которое там оказалось. Программа может работать «как-то», а потом неожиданно сломаться. Это один из самых коварных багов.

```cpp
int x;          // ⚠️ ПЛОХО: в x неизвестно что (мусор)
int y = 0;      // лучше
int z{0};       // ✅ лучше всего (современный стиль)
int n{};        // тоже хорошо: пустые скобки = ноль по умолчанию
```

🤖 Скобочки `{ }` называются **универсальная инициализация**. У них есть супер-сила: они **защищают от потери данных**.

```cpp
int a = 3.14;   // ⚠️ молча отрежет дробь, станет 3 — легко не заметить ошибку
int b{3.14};    // ✅ компилятор СКАЖЕТ «стоп, ты теряешь дробную часть!»
```

✅ **Правило Боба №1:** Каждая переменная при создании должна получать значение. Предпочитай скобки `{ }` — они ловят ошибки потери данных.

🎯 **Попробуй сам:** создай `int` через `{ }` и попробуй положить туда `2.99`. Прочитай, что скажет компилятор.

## Урок A.2. `const` — обещание «не менять»

🧠 Слово `const` говорит: «эта переменная **не должна меняться**». Если ты случайно попробуешь её изменить — компилятор остановит тебя.

```cpp
const int max_lives{3};      // правило игры: жизней всегда 3
max_lives = 5;               // ⚠️ ОШИБКА компиляции — и это хорошо!
```

🤖 Зачем себя ограничивать? Потому что **ограничения защищают от ошибок**. Если значение по смыслу не должно меняться — пометь его `const`, и компилютер будет следить за этим вместо тебя. Это называется **const-корректность**.

✅ **Правило Боба №2:** Если значение не должно меняться — делай его `const`. Лучше пусть программа не соберётся сейчас, чем сломается у пользователя потом.

## Урок A.3. Ссылки против копий 🔗

🧠 Очень важная идея. Когда ты передаёшь данные в функцию, по умолчанию делается **копия**.

```cpp
void spoil(int number) {      // number — это КОПИЯ
    number = 999;
}

int main() {
    int x{5};
    spoil(x);
    std::cout << x;   // выведет 5! Оригинал не изменился — меняли копию
}
```

А **ссылка** (`&`) — это не копия, а «второе имя» того же самого:

```cpp
void change(int& number) {    // number — ССЫЛКА на оригинал
    number = 999;
}

int main() {
    int x{5};
    change(x);
    std::cout << x;   // выведет 999! Меняли сам оригинал
}
```

🤖 А ещё копировать большие вещи (например длинный текст) — **медленно**. Поэтому большие данные передают по ссылке. Но если менять их не нужно — добавляют `const`, чтобы случайно не испортить:

```cpp
// ✅ идеально: не копируем (быстро) и не можем испортить (безопасно)
void greet(const std::string& name) {
    std::cout << "Hello, " << name << "!\n";
}
```

✅ **Правило Боба №3:** Маленькие числа (`int`, `double`) передавай как есть. Большие данные (строки, контейнеры) передавай через `const&` — быстро и безопасно. Через `&` без `const` — только когда вправду нужно изменить оригинал.

📌 Подробно про **ссылки и указатели вместе** (отличия, `nullptr`, `*`/`&`/`->`, что когда выбирать) — в файле [`Курс_C++_Указатели_и_ссылки.md`](Курс_C++_Указатели_и_ссылки.md).

## Урок A.4. Область видимости и время жизни ⏳

🧠 У каждой переменной есть **место рождения** и **место смерти**. Они «живут» внутри своих фигурных скобок `{ }`.

```cpp
int main() {
    int x{10};          // x рождается
    {
        int y{20};      // y рождается
        std::cout << x + y;   // здесь живы и x, и y → 30
    }                   // ⚰️ y УМИРАЕТ здесь (вышли из его блока)

    std::cout << y;     // ⚠️ ОШИБКА: y больше не существует
}                       // ⚰️ x умирает здесь
```

🤖 Это супер-важно запомнить — на этой идее построено **главное правило этого файла: RAII** (Часть D). «Переменная умирает, когда выходит из своих скобок» — держи эту мысль наготове.

✅ **Правило Боба №4:** Создавай переменные как можно **ближе** к месту, где они нужны, и в самом узком блоке `{ }`. Чем короче жизнь переменной, тем меньше шансов ошибиться.

🕵️ **Связь с реверсом:** «время жизни» — это про то, когда память выделяется и освобождается. Реверсеры внимательно следят за этим, изучая, как программа работает с памятью.

## Урок A.5. `auto` — пусть компилятор подумает за тебя

🧠 Иногда тип переменной длинный и очевидный. Слово `auto` говорит: «компилятор, выведи тип сам».

```cpp
auto x{5};            // компилятор понимает: это int
auto pi{3.14};        // это double
auto name{std::string{"Bob"}};   // это std::string
```

🤖 `auto` особенно полезен с длинными типами из контейнеров (увидишь в Части F). Но не злоупотребляй: иногда явный тип читается понятнее.

✅ **Правило Боба №5:** Используй `auto`, когда тип и так очевиден или слишком длинный. Если читателю кода важно видеть точный тип — пиши его явно. Главное — **понятность**.

# Часть B. Свои типы данных: `struct` 📦📦📦

## Урок B.1. Склеиваем переменные вместе

🧠 Часто несколько данных описывают **один предмет**. Например, у точки есть `x` и `y`. У игрока — имя, здоровье, очки. Держать их отдельными переменными неудобно. **Структура** (`struct`) склеивает их в один новый тип.

```cpp
struct Point {
    int x{0};
    int y{0};
};

int main() {
    Point t{3, 5};          // создаём точку
    std::cout << t.x << ", " << t.y << "\n";   // 3, 5
    t.y = 10;               // меняем одно поле
}
```

🤖 `t.x` читается «икс у точки t». Точка `.` — это «загляни внутрь». Поля (`x`, `y`) — это переменные **внутри** структуры. Заметь: я сразу написал `{0}` у полей — теперь точка по умолчанию в начале координат, никакого мусора (Правило №1!).

🎯 **Попробуй сам:** опиши `struct Player` с полями `name` (`std::string`), `health` (`int`), `score` (`int`). Создай игрока и выведи его данные.

## Урок B.2. Много структур сразу

🧠 Структуры отлично складываются в контейнеры (про которые скоро):

#include <vector>
std::vector<Point> path;         // список точек — например, маршрут
path.push_back({0, 0});
path.push_back({3, 5});

# Часть C. Небольшой экскурс в ООП 🏛️

## Урок C.1. Что такое ООП простыми словами

🧠 **ООП** = объектно-ориентированное программирование. Идея простая и красивая:

> Объединить **данные** и **действия над ними** в один «предмет» (объект).

📌 У ООП есть **четыре кита**: абстракция, инкапсуляция, наследование, полиморфизм. Короткая сводка всех четырёх вместе (с мини-кодом) — в [`Курс_C++_ООП_продвинуто.md`](Курс_C++_ООП_продвинуто.md), Часть 0. Здесь, в Части C, мы подробно разберём **инкапсуляцию**.

🤖 Сравни с реальным миром. Собака — это **данные** (имя, возраст, порода) **+ действия** (лаять, бежать, есть). Собака «знает» свои данные и «умеет» свои действия. В ООП мы описываем такие «предметы» прямо в коде. Описание предмета называется **класс**, а конкретный предмет — **объект**.

| Реальный мир | C++ |
|---|---|
| Чертёж / порода «такса» | **класс** (`class`) |
| Конкретная собака Рекс | **объект** (переменная этого класса) |
| Имя, возраст | **поля** (данные) |
| Лаять, бежать | **методы** (функции внутри класса) |

---

## Урок C.2. Первый класс

🧠 `class` похож на `struct`, но обычно содержит и данные, и методы (функции), которые с этими данными работают.

```cpp
class Counter {
public:
    void increment() { ++value_; }            // метод: увеличить на 1
    void reset()     { value_ = 0; }          // метод: сбросить
    int get() const { return value_; }        // метод: узнать значение

private:
    int value_{0};   // поле (данные), спрятано внутри
};

int main() {
    Counter s;            // создаём объект-счётчик
    s.increment();
    s.increment();
    s.increment();
    std::cout << s.get();   // 3
}
```

🤖 Разбор:
- `s.increment()` — «попроси счётчик `s` увеличиться». Это **вызов метода**.
- Метод `get() const` помечен `const` — он только **читает** данные, ничего не меняет (Правило №2 в действии!).
- Поле названо `value_` с подчёркиванием в конце — это частая договорённость, чтобы было видно «это поле класса». Не обязательно, но удобно.

## Урок C.3. Инкапсуляция: `private` и `public` 🔒

🧠 Заметил слова `public:` и `private:`?

- `public` — «**открыто** для всех». Этим можно пользоваться снаружи.
- `private` — «**спрятано** внутри». Снаружи доступа нет.

```cpp
Counter s;
s.increment();      // ✅ можно: метод public
s.value_ = 100;     // ⚠️ ОШИБКА: поле private, снаружи нельзя
```

🤖 Зачем прятать данные? Чтобы **защитить правила**. Представь класс «здоровье игрока»: оно не должно быть меньше 0 или больше 100. Если поле открыто — кто угодно случайно запишет туда `-50` или `99999`, и игра сломается. А если спрятать поле и пускать к нему **только через метод**, метод проследит за правилами:

```cpp
class Player {
public:
    Player(std::string name, int health)      // конструктор (см. след. урок)
        : name_{name}, health_{health} {}

    void takeDamage(int damage) {
        health_ -= damage;
        if (health_ < 0) health_ = 0;          // 🛡️ правило: не ниже 0
    }
    void heal(int hp) {
        health_ += hp;
        if (health_ > 100) health_ = 100;      // 🛡️ правило: не выше 100
    }

    bool isAlive() const { return health_ > 0; }
    int health() const { return health_; }
    const std::string& name() const { return name_; }

private:
    std::string name_;
    int health_{100};
};
```

✅ **Правило Боба №6:** Прячь данные в `private`, открывай наружу только методы (`public`). Тогда твой класс сам следит, чтобы данные всегда оставались правильными. Это и есть **инкапсуляция** — главная идея ООП.

🕵️ **Связь с реверсом:** инкапсуляция — это про **защиту**. А реверсеры как раз изучают, насколько хорошо данные защищены. Понимая, как строить защиту, ты понимаешь, как её исследуют.

## Урок C.4. Конструктор — правильное «рождение» объекта

🧠 **Конструктор** — это специальный метод, который вызывается **автоматически** при создании объекта. Его задача — сделать так, чтобы объект родился **сразу в правильном состоянии** (помнишь Правило №1 про инициализацию? Конструктор — его старший брат).

```cpp
class Player {
public:
    Player(std::string name, int health)
        : name_{name}, health_{health}        // ← список инициализации
    {
        std::cout << "Player " << name_ << " entered the game\n";
    }
    // ...
private:
    std::string name_;
    int health_;
};

int main() {
    Player bob{"Bob", 100};  // вызывается конструктор: bob сразу готов к игре
}
```

🤖 Часть после двоеточия (`: name_{name}, health_{health}`) называется **список инициализации**. Это правильный способ задать поля — каждое поле получает значение **сразу при рождении**, без мусора и лишних шагов.

✅ **Правило Боба №7:** Конструктор должен оставлять объект в полностью готовом, правильном состоянии. После создания объектом сразу можно пользоваться — никаких «забыл настроить».

## Урок C.5. Деструктор — объект убирает за собой 🧹

🧠 У рождения есть пара — **деструктор**. Он вызывается **автоматически**, когда объект умирает (помнишь Урок A.4 — переменная умирает на выходе из своих `{ }`). Имя деструктора — это имя класса со значком `~`.

```cpp
class Lamp {
public:
    Lamp()  { std::cout << "💡 Lamp on\n"; }    // конструктор
    ~Lamp() { std::cout << "🌑 Lamp off\n"; }   // деструктор
};

int main() {
    std::cout << "Entering the room\n";
    {
        Lamp l;                           // конструктор: лампа включилась
        std::cout << "Room is bright\n";
    }   // ⚰️ l умирает -> деструктор -> лампа ВЫКЛЮЧИЛАСЬ САМА
    std::cout << "Left the room\n";
}
```

Вывод:
```
Entering the room
💡 Lamp on
Room is bright
🌑 Lamp off
Left the room
```

🤖 Заметь: мы **нигде не написали «выключи лампу»** — она выключилась сама, автоматически, ровно когда объект `l` умер. Это не магия, это деструктор. И это — ключ к самому важному правилу C++, которое идёт следующим.

# Часть D. RAII — золотое правило C++ 🏆

> ⚠️ Это, возможно, **самая важная идея всего C++**. Звучит страшно (RAII), но смысл — простой и красивый.

## Урок D.1. Что такое «ресурс»

🧠 **Ресурс** — это что-то, что нужно сначала **взять**, а потом обязательно **вернуть**:
- кусок **памяти** (взял → освободи),
- открытый **файл** (открыл → закрой),
- соединение с интернетом (подключился → отключись).

🤖 Главная беда: легко **забыть вернуть**. Открыл файл и забыл закрыть. Взял память и забыл освободить. Такие забытые ресурсы накапливаются — это называется **утечка** (*leak*). Программа толстеет, тормозит и в итоге падает.

---

## Урок D.2. Как было плохо (и почему)

🧠 Сначала напомним указатели (подробно — в Уроке 4.5 основного курса). Память можно взять вручную словом `new` и вернуть словом `delete`:

```cpp
int* p = new int{5};   // взяли память под число
std::cout << *p;       // пользуемся (звёздочка = «значение по адресу»)
delete p;              // ⚠️ ОБЯЗАТЕЛЬНО вернуть! Забыл — утечка памяти
```

Проблема: между `new` и `delete` легко всё испортить:
```cpp
int* p = new int{5};
if (somethingWentWrong) {
    return;            // 💥 вышли из функции и ЗАБЫЛИ delete — утечка!
}
delete p;
```

🤖 На больших программах такие забытые `delete` — источник тысяч багов. Люди годами с этим мучились. А потом придумали RAII.

---

## Урок D.3. Идея RAII

🧠 **RAII** расшифровывается как *Resource Acquisition Is Initialization* — «получение ресурса есть инициализация». По-русски проще так:

> **Взял ресурс в конструкторе — верни в деструкторе.**

То есть мы делаем объект-«хозяина» ресурса. Когда объект рождается (конструктор) — он берёт ресурс. Когда умирает (деструктор) — **сам, автоматически** возвращает. А умирает он гарантированно — на выходе из `{ }` (Урок A.4 и A.5)!

🤖 Аналогия Боба: представь **библиотечную книгу с волшебством**. Ты заходишь в комнату и берёшь книгу (конструктор). Что бы ни случилось — ты вышел спокойно, или выбежал в панике, или комната загорелась — на выходе из комнаты книга **сама** возвращается в библиотеку (деструктор). Забыть вернуть **невозможно**. Вот это и есть RAII. Помнишь лампу из Урока C.5, которая выключилась сама? Это был RAII.

---

## Урок D.4. Умные указатели — RAII для памяти

🧠 Вместо опасных `new`/`delete` в современном C++ используют **умные указатели**. Они сами вызывают `delete`, когда умирают. Главный — `std::unique_ptr` (нужен `#include <memory>`).

```cpp
#include <memory>

void example() {
    auto p = std::make_unique<int>(5);   // взяли память
    std::cout << *p;                     // пользуемся, как обычным указателем
    if (somethingWentWrong) {
        return;     // ✅ ничего не забыли! Память вернётся сама
    }
}   // ⚰️ p умирает здесь -> память освобождается АВТОМАТИЧЕСКИ. delete не нужен!
```

🤖 Видишь? Никакого `delete`. Невозможно забыть. `std::make_unique<int>(5)` — это «дай мне умного хозяина для числа 5». Когда хозяин (`p`) выходит из области видимости — он убирает за собой.

> Есть ещё `std::shared_ptr` — для случаев, когда у ресурса несколько «хозяев». Он, а также **наследование, `virtual` и полиморфизм** подробно разобраны в файле `Курс_C++_ООП_продвинуто.md` (с запускаемыми примерами про гараж и технику).

---

## Урок D.5. RAII уже повсюду! 🎉

🧠 Самое приятное: ты уже пользовался RAII, даже не зная об этом. Эти штуки **сами убирают за собой**:

- `std::string` — сама управляет памятью под текст,
- `std::vector` (и другие контейнеры) — сами управляют памятью под элементы,
- `std::fstream` (файлы) — сам закрывает файл, когда умирает,
- `std::unique_ptr` — сам освобождает память.

Поэтому в правильном современном C++ слова `new` и `delete` почти **не встречаются** — за тебя всё делают RAII-объекты.

✅ **Правило Боба №8 (золотое):** **Не пиши `new`/`delete` вручную.** Нужна коллекция — бери контейнер (`std::vector`). Нужен один объект в памяти — бери `std::make_unique`. Нужен текст — `std::string`. Пусть ресурсы убирают за собой сами. Это RAII.

🔗 А **как самому написать RAII-обёртку** для «сырого» ресурса (файл, сокет, дескриптор) — правильно выделить в конструкторе, освободить в деструкторе и не нарваться на двойное освобождение — подробно в файле `Курс_C++_RAII_подробно.md`.

# Часть E. Структуры данных 🗂️

> Теперь — про то, как **организовывать** много данных. Сначала идеи (эта часть), потом готовые инструменты C++ (Часть F).

## Урок E.1. Зачем разные структуры данных

🧠 **Структура данных** — это способ организовать данные так, чтобы с ними было удобно работать. Разные задачи — разные способы.

🤖 Аналогия: для носков — ящик, для книг — полка, для коробок — стопка, для слов и их переводов — словарь. Никто не хранит коробки в ящике для носков. В программировании так же: для каждой задачи есть **своя удобная структура**.

---

## Урок E.2. Последовательность (массив / список)

🧠 Самое простое: элементы **по порядку**, у каждого есть номер (**индекс**). Нумерация в C++ начинается с **0**!

```
Индекс:   0     1     2     3
Данные: [ 10 ][ 25 ][ 7  ][ 42 ]
```

- Доступ по индексу — **мгновенный**: «дай 2-й элемент» → сразу `7`.
- Хорош, когда нужен порядок и доступ по номеру.

⚠️ Индекс первого элемента — **0**, а не 1. Это частая ошибка новичков. В массиве из 4 элементов индексы: 0, 1, 2, 3.

---

## Урок E.3. Стек (Stack) — стопка коробок 📦

🧠 **Стек** работает по принципу **LIFO**: «последним пришёл — первым ушёл» (*Last In, First Out*). Как стопка коробок на складе: кладёшь сверху, и берёшь тоже сверху.

```
   положить ↓   ↑ взять
        ┌────────┐
        │   3    │ ← верх (берётся первым)
        │   2    │
        │   1    │ ← низ
        └────────┘
```

🤖 Где встречается: кнопка «Отменить» (Ctrl+Z) — отменяет **последнее** действие. История браузера «Назад». А ещё — сам процессор использует стек для вызова функций! 🕵️ Это **очень** важно для реверса (запомни слово «стек», в Части 5 основного курса оно вернётся).

---

## Урок E.4. Очередь (Queue) — очередь в магазине 🛒

🧠 **Очередь** работает по принципу **FIFO**: «первым пришёл — первым ушёл» (*First In, First Out*). Как живая очередь: кто первый встал, того первым и обслужат.

```
  выход ←  [ A ][ B ][ C ]  ← вход
         (A первым пришёл — A первым уйдёт)
```

🤖 Где встречается: очередь печати документов, задачи на обработку, сообщения в чате по порядку.

---

## Урок E.5. Словарь / карта (Map) — ключ → значение 📖

🧠 **Словарь** (он же «карта», *map*) хранит **пары**: ключ → значение. Как настоящий словарь: слово → перевод.

```
  "Bob"   →  100
  "Anna"  →  150
  "Igor"  →  90
```

- Ищем не по номеру, а по **ключу**: «сколько очков у Anna?» → `150`.
- Идеально, когда у каждого значения есть «имя» (ключ).

🤖 Где встречается: телефонная книга (имя → номер), счёт игроков (имя → очки), настройки (название → значение).

---

## Урок E.6. Множество (Set) — без повторов 🎯

🧠 **Множество** хранит элементы **без повторений**. Добавил один и тот же элемент дважды — он останется в одном экземпляре.

```
  insert(3), insert(1), insert(3), insert(2)
  →  множество: { 1, 2, 3 }   (второй 3 не добавился)
```

🤖 Где встречается: список уникальных посетителей, набор пройденных уровней, проверка «видели ли мы уже это».

---

## Урок E.7. А ещё бывают деревья и графы 🌳

🧠 Это структуры посложнее, на будущее:
- **Дерево** — данные ветвятся, как дерево. Пример из жизни: **папки и файлы** на компьютере (папка содержит папки и файлы).
- **Граф** — точки, соединённые связями. Пример: карта метро (станции и пути), друзья в соцсети.

🤖 Глубоко мы их разберём позже, когда будем дорабатывать курс. Сейчас важно просто знать, что они есть.

✅ **Правило Боба №9:** Перед тем как писать код, спроси себя: *«какая структура данных тут подходит?»* Правильно выбранная структура делает программу простой и быстрой. Неправильная — мучительной.

# Часть F. Контейнеры STL — готовые структуры данных 🧰

## Урок F.1. Что такое STL

🧠 **STL** (*Standard Template Library*) — это огромный **сундук готовых инструментов**, встроенный в C++. Там уже есть все структуры данных из Части E — написанные профессионалами, проверенные миллионами программистов, быстрые и надёжные.

✅ **Правило Боба №10:** Не изобретай велосипед. Нужна структура данных — почти всегда в STL уже есть готовая. Бери её, а не пиши свою с нуля.

---

## Урок F.2. `std::vector` — контейнер №1 ⭐

🧠 **`std::vector`** — это «умный массив», который **сам растёт**, когда добавляешь элементы. Это самый используемый контейнер в C++. Если сомневаешься, что взять — бери `vector`.

```cpp
#include <vector>
#include <iostream>

int main() {
    std::vector<int> score;      // пустой список чисел
    score.push_back(10);         // добавить в конец
    score.push_back(25);
    score.push_back(7);

    std::cout << "Total elements: " << score.size() << "\n";    // 3
    std::cout << "First: " << score[0] << "\n";                 // 10

    // пройти по всем (современный, правильный способ):
    for (int o : score) {
        std::cout << o << " ";   // 10 25 7
    }
}
```

🤖 Полезное:
- `push_back(x)` — добавить `x` в конец.
- `size()` — сколько элементов.
- `score[i]` — элемент по индексу (быстро, но **не проверяет** выход за границы).
- `score.at(i)` — то же, но **проверяет** границы и честно ругается, если вышел за край. Безопаснее.
- `for (int o : score)` — «для каждого `o` из `score`». Это **range-based for** — правильный способ перебрать контейнер.

⚠️ `score[5]` в векторе из 3 элементов — это **выход за границу**. Программа может выдать мусор или упасть. Это классическая дыра, которую ищут реверсеры. 🕵️ Хочешь безопасно — используй `.at(5)`, он честно сообщит об ошибке.

✅ **Правило Боба №11:** Забудь про старые C-массивы (`int arr[10];`). Используй `std::vector` — он безопаснее, удобнее и сам управляет памятью (RAII!).

---

## Урок F.3. `std::array` — массив фиксированного размера

🧠 Если размер известен заранее и не меняется — есть `std::array` (нужен `#include <array>`). Это безопасная замена старому C-массиву.

```cpp
#include <array>
std::array<int, 3> coords{10, 20, 30};   // ровно 3 элемента
std::cout << coords[1];                    // 20
std::cout << coords.size();                // 3
```

🤖 Когда количество элементов фиксировано (например, 3 координаты, 7 дней недели) — `std::array`. Когда количество меняется — `std::vector`.

---

## Урок F.4. `std::string` — это тоже контейнер!

🧠 Сюрприз: `std::string` — это, по сути, контейнер символов. По нему тоже можно ходить циклом:

```cpp
#include <string>
std::string word{"Friend"};
std::cout << word.size() << "\n";      // 6 символов
std::cout << word[0] << "\n";          // F

for (char c : word) {
    std::cout << c << ".";             // F.r.i.e.n.d.
}

word += " hi";                         // склейка строк
```

---

## Урок F.5. `std::map` — словарь (ключ → значение)

🧠 **`std::map`** (нужен `#include <map>`) — это структура «словарь» из Урока E.5.

```cpp
#include <map>
#include <string>

std::map<std::string, int> score;   // ключ — имя (string), значение — очки (int)

score["Bob"] = 100;
score["Anna"] = 150;
score["Bob"] += 20;                 // у Bob стало 120

std::cout << "Anna has: " << score["Anna"] << "\n";   // 150

// пройти по всем парам (современный способ — structured bindings):
for (const auto& [name, point] : score) {
    std::cout << name << ": " << point << "\n";
}
```

🤖 Строчка `for (const auto& [name, point] : score)` читается так: «для каждой пары из словаря назови ключ `name`, а значение `point`». Удобно и читаемо! (Это работает в C++17 — помнишь, мы включали его в начале файла?)

> Есть ещё `std::unordered_map` — тот же словарь, но **быстрее** ищет (правда, не хранит ключи по порядку). Про разницу — чуть ниже, в части про скорость.

## Урок F.6. `std::set` — множество без повторов

```cpp
#include <set>

std::set<int> numbers;
numbers.insert(3);
numbers.insert(1);
numbers.insert(3);   // повтор — не добавится

// в множестве сейчас: {1, 3} — без повторов и по возрастанию
for (int n : numbers) std::cout << n << " ";    // 1 3

// проверить, есть ли элемент:
if (numbers.count(3) > 0) std::cout << "3 exists!";
```

🤖 `set` сам выкидывает повторы и держит элементы по порядку. Идеально для «уникальных» вещей.

---

## Урок F.7. `std::pair` — склейка двух значений

```cpp
#include <utility>
std::pair<std::string, int> player{"Bob", 100};
std::cout << player.first << " - " << player.second;   // Bob - 100
```

🤖 `first` и `second` — две части пары. `map` внутри как раз состоит из таких пар. Когда нужно вернуть из функции **два** значения сразу — `pair` выручает.

## Урок F.8. `std::stack` и `std::queue`

🧠 Это структуры из Уроков E.3 и E.4, готовые к использованию.

```cpp
#include <stack>
std::stack<int> s;
s.push(1); s.push(2); s.push(3);
std::cout << s.top();   // 3 — верхний
s.pop();                // убрали 3
std::cout << s.top();   // 2

#include <queue>
std::queue<int> q;
q.push(1); q.push(2); q.push(3);
std::cout << q.front(); // 1 — первый в очереди
q.pop();                // убрали 1
std::cout << q.front(); // 2
```

## Урок F.9. Шпаргалка: какой контейнер выбрать

| Что нужно | Бери | Заголовок |
|---|---|---|
| Список, который растёт | `std::vector` ⭐ | `<vector>` |
| Список фиксированного размера | `std::array` | `<array>` |
| Текст | `std::string` | `<string>` |
| Пары «ключ → значение» | `std::map` / `std::unordered_map` | `<map>` |
| Уникальные элементы | `std::set` / `std::unordered_set` | `<set>` |
| Стопка (LIFO) | `std::stack` | `<stack>` |
| Очередь (FIFO) | `std::queue` | `<queue>` |
| Склеить 2 значения | `std::pair` | `<utility>` |

✅ **Правило Боба №12:** Сомневаешься — бери `std::vector`. Нужны пары ключ-значение — `std::map`. Нужна уникальность — `std::set`. Этих трёх хватает на 90% задач.

# Часть G. Чуть-чуть алгоритмов 🧮

## Урок G.1. Что такое алгоритм

🧠 **Алгоритм** — это чёткий пошаговый план решения задачи. «Отсортировать по возрастанию», «найти самый большой», «найти нужный элемент» — всё это алгоритмы.

🤖 Хорошая новость: самые нужные алгоритмы **уже написаны** в STL (заголовок `<algorithm>`). Не нужно изобретать сортировку самому — есть `std::sort`.

---

## Урок G.2. Маленькое отступление: итераторы 🔖

🧠 Чтобы пользоваться алгоритмами, нужно знать про **итераторы**. Итератор — это как **закладка** в книге, которая показывает на элемент контейнера.

- `v.begin()` — закладка на **первый** элемент.
- `v.end()` — закладка **сразу за последним** (не на последний, а за ним!).

🤖 Многие алгоритмы работают с парой «от begin до end» — то есть «по всему контейнеру». Пока просто запомни шаблон `v.begin(), v.end()` = «весь контейнер».

---

## Урок G.3. Полезные готовые алгоритмы

```cpp
#include <algorithm>
#include <numeric>     // для accumulate
#include <vector>
#include <iostream>

int main() {
    std::vector<int> v{5, 2, 8, 1, 9};

    // отсортировать по возрастанию:
    std::sort(v.begin(), v.end());            // {1, 2, 5, 8, 9}

    // найти элемент:
    auto it = std::find(v.begin(), v.end(), 8);
    bool found = (it != v.end());             // true — 8 есть

    // посчитать, сколько раз встречается:
    int howMany = std::count(v.begin(), v.end(), 5);   // 1

    // найти максимум и минимум:
    auto maxIt = std::max_element(v.begin(), v.end()); // *maxIt == 9
    auto minIt = std::min_element(v.begin(), v.end()); // *minIt == 1

    // сумма всех элементов:
    int sum = std::accumulate(v.begin(), v.end(), 0); // 25

    std::cout << "Max: " << *maxIt << ", sum: " << sum << "\n";
}
```

✅ **Правило Боба №13:** Прежде чем писать цикл руками — проверь, нет ли готового алгоритма в `<algorithm>`. Готовый алгоритм короче, понятнее и почти всегда без ошибок.

---

## Урок G.4. Лямбды — мини-функции на месте

🧠 Иногда алгоритму нужно объяснить «как именно» что-то делать. Для этого есть **лямбда** — крошечная безымянная функция прямо в месте вызова.

```cpp
std::vector<int> v{5, 2, 8, 1, 9};

// отсортировать по УБЫВАНИЮ — объясняем лямбдой:
std::sort(v.begin(), v.end(), [](int a, int b) {
    return a > b;           // "a должно идти раньше b, если a больше"
});
// теперь v = {9, 8, 5, 2, 1}
```

🤖 `[](int a, int b){ ... }` — это и есть лямбда. Квадратные скобки `[]` — её «лицо». Не пугайся, если пока непонятно — это тема, к которой мы вернёмся подробнее. Просто знай, что так алгоритму можно задавать своё правило.

---

## Урок G.5. Чуть про скорость: почему выбор структуры важен ⚡

🧠 Представь, что ищешь имя в **телефонной книге на 1000 страниц**:
- **Способ 1:** листать подряд страницу за страницей. В худшем случае — 1000 проверок. Это «медленный» поиск (учёные называют его **O(n)** — время растёт вместе с количеством данных).
- **Способ 2:** книга отсортирована по алфавиту — открываешь середину, понимаешь, левее или правее, отбрасываешь половину, и так далее. Всего ~10 проверок на 1000 имён! Это **бинарный поиск** (**O(log n)** — очень быстро). Помнишь игру «угадай число», где выгодно говорить середину? Это тот же приём.
- **Способ 3:** `std::unordered_map` умеет находить почти **мгновенно**, за один шаг (**O(1)**), как будто сразу знает нужную страницу.

🤖 Вот зачем мы учили структуры данных! **Правильная структура = быстрая программа.** Искать в `std::vector` подряд — медленно. Искать в `std::map`/`std::unordered_map` по ключу — быстро. Эта «скорость роста» называется **алгоритмическая сложность** (или **Big O**). Глубоко разберём позже — пока достаточно интуиции: *«одни способы масштабируются лучше других»*.

✅ **Правило Боба №14:** Сначала пиши **понятно и правильно**. Потом, если надо быстрее — подумай, та ли структура данных выбрана. Чаще всего скорость берётся именно из правильной структуры, а не из «хитрого» кода.

# Часть H. Шаблоны — один код для разных типов 🧩🔁

> Самая «взрослая» тема файла, но возьмём её **просто**. Шаблоны — это идея, на которой построена вся STL (`std::vector`, `std::map`, `std::sort` — все шаблоны).

## Урок H.1. Проблема: копировать код ради другого типа

🧠 Допустим, нужна функция «вернуть большее из двух». Для `int`:

```cpp
int maximum(int a, int b) {
    if (a > b) return a;
    else       return b;
}
```

А если понадобятся `double`? Придётся писать почти то же самое ещё раз:

```cpp
double maximum(double a, double b) {   // копипаста! отличается только тип
    if (a > b) return a;
    else       return b;
}
```

🤖 Копировать код ради смены типа — скучно и неправильно (мы не любим повторяться, помнишь Правило про DRY?). Должен быть способ написать **один раз для любого типа**. Он есть — это **шаблон**.

## Урок H.2. Шаблон функции

🧠 Добавляем волшебную строчку `template <typename T>`. Здесь `T` — это «какой-то тип», заглушка, которую компилятор сам заменит на нужный.

```cpp
template <typename T>     // T — какой-то тип, выберем позже
T maximum(T a, T b) {
    if (a > b) return a;
    else       return b;
}
```

Теперь **одна** функция работает для всего:

```cpp
std::cout << maximum(5, 3)     << "\n";   // T = int    -> 5
std::cout << maximum(2.5, 7.1) << "\n";   // T = double -> 7.1
std::cout << maximum('a', 'z') << "\n";   // T = char   -> z
```

🤖 Ты написал функцию **один раз**, а компилятор сам сделал три версии — под `int`, `double` и `char`. Имя `T` — традиция (от *Type*); можно назвать иначе, но `T` понимают все.

🕵️ **Под капотом (реверс):** компилятор реально создаёт **отдельную функцию на каждый использованный тип**. Поэтому в собранной программе будут и `maximum` для int, и `maximum` для double — две похожие функции. Реверсеры это знают: шаблоны «размножают» код в бинарнике.

## Урок H.3. Сюрприз: ты уже пользовался шаблонами!

🧠 Вспомни Часть F:

```cpp
std::vector<int>            numbers;  // вектор ИНТов
std::vector<std::string>    words;    // вектор СТРОК
std::map<std::string, int>  score;    // словарь строка -> инт
```

Те самые **угловые скобки `< >`** — это и есть шаблоны! `std::vector` — шаблон класса, а `<int>` говорит ему: «храни int». Один и тот же `vector` умеет хранить что угодно именно потому, что он **шаблонный**.

🤖 То есть шаблоны были рядом с самого начала. Теперь ты понимаешь, как это устроено внутри.

## Урок H.4. Шаблон класса — своя «коробка» для любого типа

🧠 Свой класс тоже можно сделать шаблонным. Сделаем простую коробку, которая хранит одно значение **любого** типа:

```cpp
template <typename T>
class Box {
public:
    Box(T value) : value_{value} {}
    T    get() const          { return value_; }
    void put(T newValue)      { value_ = newValue; }
private:
    T value_;
};
```

Используем с разными типами:

```cpp
Box<int>         k1{42};
Box<std::string> k2{"S3CRET"};

std::cout << k1.get() << "\n";   // 42
std::cout << k2.get() << "\n";   // S3CRET
```

🤖 Одна «коробка» — и для чисел, и для строк, и для чего угодно. Примерно так (только намного умнее) и сделан `std::vector`.

## Урок H.5. Несколько типов сразу

🧠 Типов-заглушек может быть несколько:

```cpp
template <typename A, typename B>
void showPair(A first, B second) {
    std::cout << first << " : " << second << "\n";
}

showPair("port", 443);       // строка и int    -> port : 443
showPair("ip", "10.0.0.1");  // строка и строка -> ip : 10.0.0.1
```

🤖 Именно так устроены `std::pair<A, B>` (пара) и `std::map<K, V>` (ключ K → значение V) — у них по два типа-параметра.

## Урок H.6. Зачем это нужно

✅ **Правило Боба №15:** Нужен **один и тот же код для разных типов** — сделай шаблон (`template`), а не копию. Но не усложняй: шаблоны хороши там, где код вправду используется с разными типами. Если тип всегда один — обычной функции достаточно.

🤖 Шаблоны — это «верхушка айсберга»: дальше есть штуки помощнее (специализация, концепты), но они пока не нужны. Главное ты понял: **шаблон = один код, много типов**. На этой идее стоит вся стандартная библиотека.

🎯 **Попробуй сам:**
1. Напиши шаблонную функцию `summa(T a, T b)`, которая возвращает `a + b`. Проверь на `int` и на `double`.
2. Заведи `Box<int>`, положи число, поменяй его через `put`, выведи.
3. Подумай: почему `std::vector<int>` и `std::vector<double>` — это «один шаблон, два разных типа»?

# Часть I. Собираем всё вместе 🧩

🛠 **Мини-проект: «Таблица рекордов».** Он использует почти всё из этого файла: `struct`, `std::vector`, алгоритм `std::sort`, лямбду, `const&`, `{ }`-инициализацию и range-based for. И всё это — правильно и без единого `new`/`delete` (RAII работает сам).

```cpp
#include <iostream>
#include <vector>
#include <string>
#include <algorithm>

// свой тип данных: одна строчка таблицы рекордов
struct Record {
    std::string name;
    int score{0};      // инициализирован — никакого мусора (Правило №1)
};

int main() {
    // вектор структур — наша таблица
    std::vector<Record> table{
        {"Bob", 120},
        {"Anna", 200},
        {"Igor", 90},
        {"Sveta", 200},
    };

    // сортируем по убыванию очков (алгоритм + лямбда):
    std::sort(table.begin(), table.end(),
              [](const Record& a, const Record& b) {   // const& — не копируем (Правило №3)
                  return a.score > b.score;
              });

    std::cout << "=== RECORDS TABLE ===\n";
    int place{1};
    for (const Record& r : table) {       // range-for + const&
        std::cout << place << ". " << r.name << " - " << r.score << "\n";
        ++place;
    }
    return 0;
}
```

Вывод:
```
=== RECORDS TABLE ===
1. Anna - 200
2. Sveta - 200
3. Bob - 120
4. Igor - 90
```

🎯 **Попробуй сам (по нарастающей):**
1. Добавь ещё трёх игроков в таблицу.
2. Выведи отдельной строкой имя **лучшего** игрока (подсказка: после сортировки это `tablica[0]`, но безопаснее проверить, что таблица не пустая, и использовать `.front()`).
3. Сделай так, чтобы программа **спрашивала** игроков и их очки у пользователя (`std::cin`) и добавляла в таблицу через `push_back`.
4. Заведи `std::map<std::string, int>`, посчитай в нём суммарные очки, если один игрок встречается несколько раз.

## 📖 Словарик (новые слова)

| Слово | Простыми словами |
|---|---|
| **Инициализация** | Задать значение переменной сразу при создании |
| **`const`** | Пометка «это нельзя менять» |
| **Ссылка (`&`)** | Второе имя той же переменной (не копия) |
| **Область видимости** | Где переменная «живёт» (внутри своих `{ }`) |
| **`struct` / `class`** | Свой составной тип данных |
| **Поле** | Переменная внутри структуры/класса |
| **Метод** | Функция внутри класса |
| **Объект** | Конкретный экземпляр класса |
| **Инкапсуляция** | Прятать данные, открывать только методы |
| **Конструктор** | Метод-«рождение» объекта |
| **Деструктор** | Метод-«уборка», вызывается при смерти объекта |
| **RAII** | Взял ресурс в конструкторе — вернул в деструкторе (авто-уборка) |
| **Умный указатель** | Указатель, который сам освобождает память |
| **STL** | Стандартная библиотека контейнеров и алгоритмов |
| **Контейнер** | Готовая структура данных (`vector`, `map`, `set`…) |
| **Итератор** | «Закладка» на элемент контейнера |
| **Алгоритм** | Готовый способ-инструкция (`sort`, `find`…) |
| **Лямбда** | Маленькая безымянная функция на месте |
| **Шаблон (template)** | Один код, работающий для любого типа (`<T>`) |
| **Обобщённый код** | Код, не привязанный к конкретному типу |
| **Сложность / Big O** | Как быстро растёт время работы с ростом данных |
