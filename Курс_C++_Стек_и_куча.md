# 🤖📦 Курс C++ с Бобом — Стек или куча: где создавать объекты

> Ты уже знаешь, что у программы есть **стек** и **куча** (почему так — в `Курс_C++_Под_капотом.md`, Урок 3.4). Теперь — **самый практический вопрос**: когда пишешь код, как положить объект на стек, а как в кучу, и **как понять, где и для чего** его создавать. Максимально просто, с примерами, которые можно запустить.

**Когда читать:** после `Курс_C++_Под_капотом.md` (карта памяти) и Части D `Курс_C++_Основы_глубже.md` (RAII, `unique_ptr`).

🧠 Очень коротко напомню (подробно — в Под капотом):
- **Стек** — быстрый, **автоматический**; объект живёт внутри своих `{ }` и сам исчезает в конце. Размер ограничен.
- **Куча** — гибкая; объект живёт, **пока им владеют**; памятью надо управлять (за нас это делают **умные указатели** и **контейнеры**).

Значки: 🤖 объяснение · 🧠 теория · 🛠 практика · 🎯 задание · ⚠️ важно · ✅ правило · 🕵️ реверс.

# Часть 1. Два способа создать объект

🔧 Возьмём простой класс «машина» и создадим его двумя способами.

```cpp
#include <iostream>
#include <memory>
#include <string>

struct Car {
    std::string name;
    Car(std::string n) : name{n} {}
    void signal() const { std::cout << name << ": beep\n"; }
};

int main() {
    // 1) НА СТЕКЕ — просто объявили переменную:
    Car m1{"Sedan"};
    m1.signal();                                   // через ТОЧКУ .

    // 2) В КУЧЕ — через умный указатель:
    auto m2 = std::make_unique<Car>("Pickup");
    m2->signal();                                  // через СТРЕЛОЧКУ ->
}   // и m1, и память под m2 уберутся сами
```

**Ожидаемый вывод:**
```
Sedan: beep
Pickup: beep
```

🤖 Главное замечай:
- **Стек:** `Car m1{"Sedan"};` — обычная переменная. Доступ через **точку** `m1.signal()`.
- **Куча:** `std::make_unique<Car>("Pickup")` — объект создаётся в куче, а `m2` — «умный хозяин» на него. Доступ через **стрелочку** `m2->signal()`.
- ⚠️ Никаких ручных `new`/`delete` — только `make_unique` (Правило RAII из Части D).

| | Стек | Куча |
|---|---|---|
| Как создать | `Car m{"..."};` | `auto m = std::make_unique<Car>("...");` |
| Доступ | через `.` | через `->` |
| Когда исчезает | в конце своих `{ }` | когда уходит последний «хозяин» |
| Управление | само | умный указатель / контейнер |

# Часть 2. Как понять, где создавать — простое правило 🧭

✅ **Правило по умолчанию: создавай на СТЕКЕ.**
Просто объяви переменную. Это быстро, безопасно и само убирается. *Сомневаешься — стек.*

🤖 А в **КУЧУ** клади объект только когда есть конкретная причина. Их всего четыре:

**Причина 1 — объект должен жить дольше функции, где создан.**
Объект на стеке умирает в конце функции. Если он нужен **после** — клади в кучу и передавай владение наружу (`unique_ptr`).

**Причина 2 — хранишь разные потомки вместе (полиморфизм).**
`std::vector<std::unique_ptr<Transport>>` — разные машины/мотоциклы в одном списке (см. капстоун в `Курс_C++_ООП_продвинуто.md`). Через базовый указатель — значит, через кучу.

**Причина 3 — объектов много или они большие.**
Стек **ограничен** (несколько мегабайт). Большой массив на стеке его переполнит. Решение — **контейнер** (`std::vector`), он сам кладёт данные в кучу.

**Причина 4 — у объекта несколько хозяев.**
Тогда `std::shared_ptr` (счётчик хозяев, см. продвинутое ООП).

🧠 Если ни одна причина не подходит — **стек**.

# Часть 3. Примеры бок о бок (можно проверить)

## Пример A. Короткоживущий объект → стек

```cpp
void check() {
    Car m{"Sedan"};   // нужен только здесь
    m.signal();
}                         // m исчезла — и хорошо
```
🤖 Объект нужен только внутри функции → стек. Ничего лишнего.

## Пример B. Должен пережить функцию → куча

```cpp
// ⚠️ ПЛОХО: вернуть указатель на объект со стека
Car* bad() {
    Car m{"Sedan"};
    return &m;            // m УМРЁТ в конце функции -> указатель станет «битым»!
}

// ✅ ХОРОШО: объект в куче переживёт функцию
std::unique_ptr<Car> assemble() {
    return std::make_unique<Car>("Sedan");   // живёт, пока им владеют
}

int main() {
    auto m = assemble();   // забрали владение объектом из кучи
    m->signal();         // Sedan: beep
}
```
🤖 В `bad` объект на стеке умирает вместе с функцией, и возвращённый указатель показывает на **пустое место** (это «битый/висячий указатель» — классический баг). В `assemble` объект в куче, и владение передаётся наружу через `unique_ptr` — он живёт дальше.

## Пример C. Много / большой объём → контейнер (он сам в куче)

```cpp
#include <vector>
std::vector<int> big(10000000);   // десять миллионов чисел — в КУЧЕ за тебя
```
🤖 Не пытайся завести гигантский массив на стеке (`int a[10000000];`) — будет **переполнение стека**. Возьми `std::vector` — он молча кладёт данные в кучу и сам ими управляет.

🎯 **Попробуй сам:** напиши функцию, которая создаёт `Car` и **возвращает** её новому хозяину. Каким способом — стек или куча? (Подсказка: объект должен пережить функцию.)

# Часть 4. Скрытая куча: ты уже ей пользуешься 🙂

🧠 Сюрприз: `std::vector`, `std::string` и другие контейнеры **сами хранят свои данные в куче**, а тебе показывают удобную «обёртку» на стеке.

```cpp
std::string name = "Bob";       // сама строка-буфер — в куче, а переменная name — на стеке
std::vector<int> points;        // элементы — в куче, vector управляет ими сам
```

🤖 Поэтому правильный современный совет звучит так: **не управляй кучей вручную**. Нужно много объектов или большой объём — бери **контейнер**. Нужен один объект в куче (полиморфизм, передать владение) — бери **`make_unique`**. Всё остальное — **стек**. Куча работает на тебя незаметно.

# Часть 5. Аллокаторы — откуда контейнер берёт память 🏭

🧠 Когда `std::vector` нужно место под элементы, он берёт память **не сам** — он просит **аллокатор**. Аллокатор — это «снабженец»: его задача **достать** кусок памяти и потом **вернуть**.

🤖 У контейнеров аллокатор — это **скрытый** параметр шаблона. `std::vector<int>` на самом деле — `std::vector<int, std::allocator<int>>`. По умолчанию это `std::allocator`, который берёт память через `operator new` — то есть из **кучи**.

🔧 Аналогия: контейнер — бригада строителей, аллокатор — **кладовщик**, выдающий материалы. По умолчанию кладовщик за каждым материалом бегает на большой склад (куча). Но его можно **заменить** на своего — например, который заранее взял целый ящик и раздаёт из него мгновенно.

### Зачем вообще менять аллокатор (что для чего)

- ⚡ **Скорость:** много мелких `new` — медленно. Свой аллокатор берёт большой кусок **один раз** и быстро раздаёт из него куски (это называют **пул** / арена). Игры, реальное время.
- 🎯 **Контроль места:** положить данные в конкретный буфер — даже **на стек**, вообще не трогая кучу.
- 🐞 **Отладка:** считать или логировать каждое выделение памяти.
- 🔧 **Особая память:** выровненная, разделяемая между процессами и т.п.

### 🛠 Пример: вектор, который кладёт данные на стек

🧠 Самый простой современный способ задать «свой источник памяти» — **`std::pmr`** (C++17). Сделаем вектор, который берёт память из **буфера на стеке**, а не из кучи:

```cpp
#include <iostream>
#include <vector>
#include <array>
#include <memory_resource>   // std::pmr (нужен C++17 и свежий компилятор)

int main() {
    std::array<std::byte, 1024> buffer;   // 1 КБ памяти прямо НА СТЕКЕ
    std::pmr::monotonic_buffer_resource pool(buffer.data(), buffer.size());

    std::pmr::vector<int> v(&pool);        // вектор берёт память из НАШЕГО буфера
    v.push_back(1);
    v.push_back(2);
    v.push_back(3);

    for (int x : v) std::cout << x << " ";   // 1 2 3
}
```

**Ожидаемый вывод:**
```
1 2 3
```

🤖 Фокус в том, что `std::pmr::vector` сложил элементы в `buffer` (а он на **стеке**) и **не сделал ни одного `new`** в кучу. Когда буфер закончится — `monotonic_buffer_resource` сам добёрет из кучи. Мы просто **сменили снабженца**: вместо «бегать на склад-кучу» он раздаёт из заранее взятого ящика. Для множества мелких выделений это **намного быстрее**.

✅ **Правило Боба:** в 99% случаев **дефолтный аллокатор — именно то, что нужно**, и думать о нём не надо. Свои аллокаторы / `pmr` достают, только когда упёрлись в скорость работы с памятью или нужен особый источник. Знать, что они есть и зачем — полезно; углубляться — по необходимости.

# Часть 6. Шпаргалка-решатель ✅

```
Нужен объект...
│
├─ только в этой функции, обычного размера?      → СТЕК:  Tip obj{...};
│
├─ должен пережить функцию (передаёшь владение)?  → КУЧА:  std::make_unique<Tip>(...)
│
├─ разные потомки в одном списке (полиморфизм)?   → КУЧА:  std::vector<std::unique_ptr<База>>
│
├─ много объектов / большой объём?                → КОНТЕЙНЕР: std::vector<Tip>  (сам в куче)
│
└─ несколько хозяев у объекта?                    → КУЧА:  std::shared_ptr<Tip>
```

## ⚠️ Частые ошибки

- Вернуть указатель/ссылку на **локальный** объект со стека → «битый указатель» (Пример B).
- Завести **огромный** объект на стеке → переполнение стека (бери `vector`).
- Писать `new`/`delete` руками → используй `make_unique`/контейнеры.
- Лезть в кучу **без причины** → она медленнее и сложнее; если хватает стека — бери стек.

🕵️ **Реверс-связь:** по адресу объекта видно, где он живёт (маленький/средний адрес — куча, очень большой — стек; см. Под капотом). А вызовы выделения кучи (`new`/`make_unique` → внутри `malloc`/`operator new`) реверсер замечает в дизассемблере — по ним он понимает, где программа берёт динамическую память.

## 📖 Словарик

| Слово | Простыми словами |
|---|---|
| **Объект на стеке** | Обычная локальная переменная; исчезает в конце `{ }` |
| **Объект в куче** | Создан через `make_unique`/контейнер; живёт, пока им владеют |
| **`.` vs `->`** | Точка — у объекта-значения; стрелочка — у указателя |
| **Висячий («битый») указатель** | Показывает на объект, который уже умер |
| **Переполнение стека** | Слишком большой/глубокий объект не влез в стек |
| **Скрытая куча** | `vector`/`string` сами держат данные в куче за тебя |
| **Аллокатор** | «Снабженец» памяти для контейнера (по умолчанию `std::allocator` → куча) |
| **`std::pmr`** | Простой способ задать свой источник памяти (например, буфер на стеке) |

---

> 🤖 Запомни одной фразой: **по умолчанию — стек; в кучу только по причине** (пережить функцию, полиморфизм, большой объём, несколько хозяев), и почти всегда через **умный указатель или контейнер**, а не вручную.
>
> Будем **дорабатывать**: вернуть объект по значению (move-семантика) vs `unique_ptr`, и почему «стек быстрее кучи».
