# Часть 0. Идеология ООП — четыре кита 🐘🐘🐘🐘

> Короткая сводка «о чём вообще ООП». Все эти идеи разобраны подробно в разных местах курса — здесь они собраны **вместе**, чтобы видеть картину целиком.

🧠 ООП стоит на **четырёх китах**:

**1. 🎚️ Абстракция — «показывай главное, прячь сложное».**
Выдели у предмета только то, что важно для задачи, а внутреннюю сложность спрячь за простым «пультом».
🔧 Пример: у машины руль, педали, рычаг — ты **управляешь**, не зная, как именно устроен двигатель. Класс так же даёт наружу простые методы, а детали прячет.

**2. 🔒 Инкапсуляция — «данные + действия вместе, и под защитой».**
Объедини данные и методы в один объект, а сами данные **спрячь** (`private`), оставив наружу только методы-«кнопки». Тогда объект сам следит, чтобы данные были правильными.
🔧 Пример: капот закрыт; до двигателя добираешься не руками, а через приборы. Подробно — [`Курс_C++_Основы_глубже.md`](Курс_C++_Основы_глубже.md), Часть C.

**3. 🌳 Наследование — «потомок берёт от базы» (is-a).**
Общее пишем один раз в базовом классе, потомки его берут и добавляют своё. *Машина **это** транспорт.*
🔧 Пример: все транспортные средства имеют колёса и умеют ехать — это в «базе»; машина и мотоцикл лишь уточняют детали. Подробно — Часть 1 ниже.

**4. 📢 Полиморфизм — «один вызов — разное поведение».**
Одна и та же команда у разных потомков срабатывает по-своему.
🔧 Пример: команда «сигналь» — у машины клаксон, у мотоцикла тонкий бип, у грузовика гудок. Зовёшь одинаково, звучит по-разному. Подробно — Часть 3 ниже.

🤖 Как связаны: **абстракция** решает, *что показать наружу*; **инкапсуляция** *прячет и защищает* внутреннее; **наследование** *переиспользует* общее; **полиморфизм** даёт потомкам *вести себя по-своему*. Вместе они превращают код в набор понятных, надёжных «предметов».

## Все четыре кита в одном мини-примере

```cpp
#include <iostream>
#include <string>

class Vehicle {
public:
    Vehicle(std::string n) : name_{n} {}
    void info() const { std::cout << name_; }      // 🎚️ АБСТРАКЦИЯ: простой «пульт» наружу
    virtual void signal() const { std::cout << ": beep\n"; }   // 📢 ПОЛИМОРФИЗМ: можно переопределить
    virtual ~Vehicle() = default;
protected:
    std::string name_;
private:
    int secretCode_ = 1234;     // 🔒 ИНКАПСУЛЯЦИЯ: спрятано, снаружи не достать
};

class Car : public Vehicle {           // 🌳 НАСЛЕДОВАНИЕ: машина это транспорт
public:
    Car(std::string n) : Vehicle{n} {}
    void signal() const override {           // 📢 своё поведение
        info(); std::cout << ": BEEEP (horn)\n";
    }
};

int main() {
    Car m{"Sedan"};
    m.signal();          // зовём как у базы — а работает версия Car
    // m.secretCode_ = 0;   // ⛔ нельзя: поле private (инкапсуляция защитила)
}
```

**Ожидаемый вывод:**
```
Sedan: BEEEP (horn)
```

🤖 В этих ~20 строках живут все четыре кита: простой метод наружу (абстракция), спрятанное поле (инкапсуляция), `Car : public Vehicle` (наследование) и `override` своего `signal()` (полиморфизм). Дальше в файле разберём наследование и полиморфизм подробно и с проверяемыми примерами.

# Часть 1. Наследование — «машина это транспорт» 🚗

## Урок 1.1. Идея наследования

🧠 **Наследование** — это когда один класс (**потомок**) берёт всё от другого (**базы**) и что-то добавляет. Связь читается как **«是 / is-a»**: *машина **это** транспорт*, *мотоцикл **это** транспорт*.

🔧 Зачем: написал общее **один раз** в базе — все потомки им пользуются. Не надо копировать.

```cpp
#include <iostream>
#include <string>

// БАЗА: любой транспорт
class Vehicle {
public:
    Vehicle(std::string name, int wheels)
        : name_{name}, wheels_{wheels} {}

    void info() const {                      // общий метод — напишем 1 раз
        std::cout << name_ << ": wheels " << wheels_ << "\n";
    }
protected:                                   // protected — видно потомкам
    std::string name_;
    int wheels_;
};

// ПОТОМОК: машина — это транспорт
class Car : public Vehicle {
public:
    Car(std::string name)
        : Vehicle{name, 4} {}                // вызываем конструктор базы
};

class Motorcycle : public Vehicle {
public:
    Motorcycle(std::string name)
        : Vehicle{name, 2} {}
};

int main() {
    Car  m{"Sedan"};
    Motorcycle mc{"Sport"};
    m.info();      // info() унаследован от базы!
    mc.info();
}
```

**Ожидаемый вывод:**
```
Sedan: wheels 4
Sport: wheels 2
```

🤖 Разбор:
- `class Car : public Vehicle` — «Car наследует от Vehicle».
- `: Vehicle{name, 4}` — потомок обязан **запустить конструктор базы** (у базы нет конструктора без аргументов).
- Метод `info()` написан в базе, а работает у всех потомков. Вот оно — переиспользование.

## Урок 1.2. `public`, `protected`, `private` — кто что видит 🔒

🧠 Три уровня доступа (помнишь `private`/`public` из Части C? добавился `protected`):

| Модификатор | Кто видит |
|---|---|
| `public` | **все** (и снаружи, и потомки) |
| `protected` | **сам класс + потомки** (но не снаружи) |
| `private` | **только сам класс** (даже потомки не видят) |

🤖 В примере выше `name_` и `wheels_` сделаны `protected` — значит, потомки (`Car`, `Motorcycle`) смогут ими пользоваться, а вот «снаружи» (`m.name_`) — нельзя. Это инкапсуляция, но «приоткрытая для своих».

# Часть 2. Конструкторы и деструкторы при наследовании 🏗️

🧠 Важный порядок: при создании объекта сначала собирается **база**, потом **потомок**. При уничтожении — наоборот. Давай **проверим это** живьём.

```cpp
#include <iostream>

class Engine {                 // двигатель (деталь внутри транспорта)
public:
    Engine()  { std::cout << "  Engine assembled\n"; }
    ~Engine() { std::cout << "  Engine disassembled\n"; }
};

class Vehicle {
public:
    Vehicle()  { std::cout << "Vehicle: frame ready\n"; }
    ~Vehicle() { std::cout << "Vehicle: frame disassembled\n"; }
private:
    Engine dv_;                // транспорт содержит двигатель
};

class Car : public Vehicle {
public:
    Car()  { std::cout << "Car: ready\n"; }
    ~Car() { std::cout << "Car: to disassembly\n"; }
};

int main() {
    std::cout << "-- assembling --\n";
    Car m;
    std::cout << "-- driving --\n";
}   // здесь m уничтожается
```

**Ожидаемый вывод:**
```
-- assembling --
  Engine assembled
Vehicle: frame ready
Car: ready
-- driving --
Car: to disassembly
Vehicle: frame disassembled
  Engine disassembled
```

🤖 Смотри порядок:
- **Сборка** идёт изнутри наружу: сначала деталь (двигатель) → потом база (рама) → потом сам потомок (машина). Логично: нельзя собрать машину раньше её рамы и двигателя.
- **Разборка** — строго наоборот: сначала машина, потом рама, потом двигатель.

🔧 Это как в гараже: сначала ставят раму и двигатель, потом собирают машину; разбирают — в обратном порядке.

# Часть 3. `virtual` и полиморфизм — «каждый сигналит по-своему» 📢

🧠 **Полиморфизм** — самая мощная идея ООП: один и тот же вызов работает **по-разному** в зависимости от того, какой объект на самом деле. Ключевое слово — **`virtual`**.

```cpp
#include <iostream>
#include <string>

class Vehicle {
public:
    Vehicle(std::string n) : name_{n} {}
    virtual void signal() const {            // virtual = «потомки могут переопределить»
        std::cout << name_ << ": beep!\n";
    }
    virtual ~Vehicle() = default;            // виртуальный деструктор (см. Часть 4!)
protected:
    std::string name_;
};

class Car : public Vehicle {
public:
    Car(std::string n) : Vehicle{n} {}
    void signal() const override {           // override = «переопределяю метод базы»
        std::cout << name_ << ": BEEEP! (horn)\n";
    }
};

class Motorcycle : public Vehicle {
public:
    Motorcycle(std::string n) : Vehicle{n} {}
    void signal() const override {
        std::cout << name_ << ": beep-beep! (thin)\n";
    }
};

void honk(const Vehicle& t) {                // работает с ЛЮБЫМ транспортом
    t.signal();                              // вызовется НУЖНАЯ версия!
}

int main() {
    Car  m{"Sedan"};
    Motorcycle mc{"Sport"};
    Vehicle t{"Cart"};
    honk(m);
    honk(mc);
    honk(t);
}
```

**Ожидаемый вывод:**
```
Sedan: BEEEP! (horn)
Sport: beep-beep! (thin)
Cart: beep!
```

🤖 Магия: функция `honk` знает только про базовый `Vehicle`, но вызывает **правильный** `signal()` для каждого объекта. Это и есть полиморфизм.
- `virtual` в базе — «разрешаю переопределять».
- `override` в потомке — «я переопределяю» (и компилятор проверит, что ты не опечатался в имени — это часть **правильности**).
- Заметь: `Car::signal` пользуется `name_` — это и есть доступ к `protected`-полю базы (Часть 1.2).

⚠️ Если убрать `virtual` — `honk(m)` всегда вызовет версию **базы** («beep»), даже для машины. `virtual` включает «выбор по реальному типу».

🎯 **Попробуй сам:** добавь класс `Truck` со своим сигналом и прогони через `honk`. Проверь, что выводится твоя версия.

# Часть 4. Виртуальный деструктор — это обязательно! ⚠️

🧠 Самое частое и опасное правило новичков. **Проверим на примере**, почему деструктор базы должен быть `virtual`.

```cpp
#include <iostream>
#include <memory>

class Vehicle {
public:
    virtual ~Vehicle() { std::cout << "~Vehicle\n"; }   // ВИРТУАЛЬНЫЙ
};

class Car : public Vehicle {
public:
    ~Car() override { std::cout << "~Car\n"; }
};

int main() {
    std::unique_ptr<Vehicle> t = std::make_unique<Car>();  // база указывает на потомка
}   // t уничтожается здесь
```

**Ожидаемый вывод (деструктор `virtual`):**
```
~Car
~Vehicle
```

⚠️ А теперь **убери слово `virtual`** у `~Transport` и запусти снова. Вывод станет:
```
~Vehicle
```
Деструктор потомка `~Car` **не вызвался**! Машина «не разобралась» — а если бы внутри `Car` была память/файл/сокет, это была бы **утечка**.

✅ **Правило Боба (железное):** если у класса есть хоть одна `virtual`-функция (значит, его будут использовать через указатель/ссылку на базу) — делай **деструктор `virtual`**. Иначе при удалении через базовый указатель потомок не приберётся за собой.

# Часть 5. Абстрактные классы и интерфейсы — «контракт» 📜

🧠 Иногда базовый класс — это просто **обещание/контракт**: «любой инструмент ОБЯЗАН уметь включаться», а как именно — решает каждый сам. Такой метод делают **чисто виртуальным** (`= 0`), а класс становится **абстрактным** — его нельзя создать напрямую.

```cpp
#include <iostream>

// Абстрактный класс — контракт для любого инструмента
class Tool {
public:
    virtual void turnOn() const = 0;     // = 0 -> чисто виртуальный, тела нет
    virtual ~Tool() = default;
};

class Drill : public Tool {
public:
    void turnOn() const override { std::cout << "Drill: bzzz, drilling\n"; }
};

class VacuumCleaner : public Tool {
public:
    void turnOn() const override { std::cout << "VacuumCleaner: whoosh, sucking\n"; }
};

int main() {
    // Tool i;          // ⛔ ОШИБКА: абстрактный класс нельзя создать
    Drill d;
    VacuumCleaner p;
    d.turnOn();
    p.turnOn();
}
```

**Ожидаемый вывод:**
```
Drill: bzzz, drilling
VacuumCleaner: whoosh, sucking
```

🤖 `= 0` означает «тела нет, реализуй сам». Каждый потомок **обязан** написать `turnOn()`. Такой класс-контракт называют **интерфейсом**. 🔧 Как розетка-стандарт: любой прибор обязан иметь вилку нужной формы, а уж что он делает внутри — его дело.

# Часть 6. Умные указатели подробно 🧠👉

🧠 В Части D углублённого файла мы познакомились с `unique_ptr`. Теперь — вся семья.

## Урок 6.1. `unique_ptr` — один хозяин

🤖 `std::unique_ptr` — **единственный** владелец объекта. Когда он уходит — объект уничтожается (RAII). Передать владение можно, но «скопировать двух хозяев» нельзя. 🔧 Как ПТС на машину: владелец один.

## Урок 6.2. `shared_ptr` — несколько хозяев

🧠 `std::shared_ptr` — когда у одного объекта **несколько** владельцев. Он считает хозяев (счётчик ссылок); объект уничтожается, когда уходит **последний**. **Проверим** счётчик:

```cpp
#include <iostream>
#include <memory>
#include <string>

class Device {                   // дорогой диагностический прибор в гараже
public:
    Device(std::string n) : n_{n} { std::cout << n_ << " bought\n"; }
    ~Device()                      { std::cout << n_ << " written off\n"; }
    void use() const               { std::cout << n_ << " in use\n"; }
private:
    std::string n_;
};

int main() {
    std::shared_ptr<Device> mechanic1 = std::make_shared<Device>("Scanner");
    std::cout << "owners: " << mechanic1.use_count() << "\n";

    {
        std::shared_ptr<Device> mechanic2 = mechanic1;   // второй механик берёт тот же прибор
        std::cout << "owners: " << mechanic1.use_count() << "\n";
        mechanic2->use();
    }   // mechanic2 ушёл -> хозяев снова 1

    std::cout << "owners: " << mechanic1.use_count() << "\n";
}   // ушёл последний хозяин -> прибор спишут
```

**Ожидаемый вывод:**
```
Scanner bought
owners: 1
owners: 2
Scanner in use
owners: 1
Scanner written off
```

🔧 Образ: один дорогой сканер на гараж, им пользуются несколько механиков. Пока хоть кто-то держит его — он жив. Списывают, только когда **последний** отпустил.

## Урок 6.3. `weak_ptr` — наблюдать, но не владеть 👀

🧠 Бывает беда: два `shared_ptr` указывают **друг на друга** (машина знает свой гараж, гараж знает свою машину). Тогда они **держат друг друга вечно** и никогда не уничтожатся — это **цикл**. Чтобы его разорвать, одну из связей делают **`weak_ptr`** — «наблюдатель, который не считается хозяином». Он смотрит на объект, но не мешает его уничтожить.

🤖 Пока просто запомни: `weak_ptr` — против **циклов** из `shared_ptr`. Подробный пример разберём, когда будем дорабатывать.

## Урок 6.4. Какой указатель выбрать

| Указатель | Владение | Когда брать |
|---|---|---|
| `unique_ptr` | один хозяин | **по умолчанию** |
| `shared_ptr` | несколько хозяев | когда владение вправду делят |
| `weak_ptr` | не владеет, наблюдает | разорвать цикл из `shared_ptr` |
| сырой `T*` | не владеет | просто «посмотреть», без управления |

✅ **Правило Боба:** по умолчанию — `unique_ptr`. `shared_ptr` — только когда у объекта реально несколько хозяев (за счётчик надо платить). Сырой `new`/`delete` — не используем (Часть D).

# Часть 7. Всё вместе: гараж полиморфных машин 🔧🚗🏍️

🧠 Соединим **наследование + `virtual` + умные указатели + контейнер**. Это **главный современный приём** C++: список умных указателей на базовый класс, в котором лежат разные потомки.

```cpp
#include <iostream>
#include <memory>
#include <vector>
#include <string>

class Vehicle {
public:
    Vehicle(std::string n) : n_{n} {}
    virtual void signal() const { std::cout << n_ << ": beep\n"; }
    virtual ~Vehicle() = default;            // virtual деструктор — обязательно!
protected:
    std::string n_;
};

class Car : public Vehicle {
public:
    Car(std::string n) : Vehicle{n} {}
    void signal() const override { std::cout << n_ << ": BEEEP\n"; }
};

class Motorcycle : public Vehicle {
public:
    Motorcycle(std::string n) : Vehicle{n} {}
    void signal() const override { std::cout << n_ << ": beep-beep\n"; }
};

int main() {
    // ГАРАЖ: список разного транспорта (умные указатели на базу)
    std::vector<std::unique_ptr<Vehicle>> garage;
    garage.push_back(std::make_unique<Car>("Sedan"));
    garage.push_back(std::make_unique<Motorcycle>("Sport"));
    garage.push_back(std::make_unique<Car>("Pickup"));

    for (const auto& t : garage) {
        t->signal();         // каждый сигналит ПО-СВОЕМУ
    }
}   // весь гараж сам уберётся: unique_ptr + virtual destructor
```

**Ожидаемый вывод:**
```
Sedan: BEEEP
Sport: beep-beep
Pickup: BEEEP
```

🤖 Здесь сошлось всё, чему мы учились:
- **наследование** — `Car`/`Motorcycle` от `Vehicle`;
- **полиморфизм** — `t->signal()` зовёт нужную версию;
- **умные указатели** — `unique_ptr` сам всё уберёт;
- **виртуальный деструктор** — потомки правильно разберутся;
- **контейнер** — `vector` хранит разнородный транспорт.

🎯 **Попробуй сам:** добавь класс `Bicycle` (сигнал «дзынь») и положи его в гараж. Запусти — убедись, что он сигналит по-своему.

# 🕵️ Реверс: как полиморфизм выглядит внутри (vtable)

🧠 Когда у класса есть `virtual`-функции, компилятор добавляет в каждый объект **скрытый указатель** на **таблицу виртуальных функций** (*vtable*) — список «какую версию метода звать». `t->signal()` под капотом = «возьми vtable объекта → найди там signal → вызови».

🕵️ Реверсеры это знают: по **vtable-указателям** они **узнают тип объекта** и **восстанавливают иерархию классов** (кто от кого наследует). Поэтому даже из «голого» бинарника можно понять структуру ООП-программы. Полиморфизм удобен для тебя — но оставляет в программе характерные следы.

---

## ✅ Мини-кодекс этого файла

| Правило |
|---|
| Наследуй, когда «потомок **это** база» (машина — это транспорт) |
| Прячь поля; для потомков — `protected`, не `public` |
| Метод переопределяешь — пиши `override` (компилятор проверит) |
| Есть `virtual`-методы → деструктор **обязательно** `virtual` |
| Контракт для потомков → чисто виртуальные методы (`= 0`) |
| По умолчанию `unique_ptr`; `shared_ptr` — только при общем владении |
| Современный приём: `vector<unique_ptr<База>>` для разнородных объектов |

## 📖 Словарик

| Слово | Простыми словами |
|---|---|
| **Наследование** | Потомок берёт всё от базы и добавляет своё |
| **База / потомок** | От кого наследуют / кто наследует |
| **is-a** | «Машина **это** транспорт» — признак наследования |
| **`protected`** | Видно классу и его потомкам, но не снаружи |
| **`virtual`** | Метод можно переопределить в потомке |
| **`override`** | Пометка «я переопределяю метод базы» |
| **Полиморфизм** | Один вызов — разное поведение по реальному типу |
| **Виртуальный деструктор** | Чтобы потомок правильно убрался через базовый указатель |
| **Абстрактный класс** | Класс-контракт с `= 0`; нельзя создать напрямую |
| **Интерфейс** | Класс из одних чисто виртуальных методов (обещание) |
| **`shared_ptr`** | Умный указатель с несколькими хозяевами (счётчик) |
| **`weak_ptr`** | Наблюдатель без владения; разрывает циклы |
| **vtable** | Скрытая таблица «какой метод звать»; след ООП в бинарнике |
