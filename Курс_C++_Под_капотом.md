# Часть 1. Путь программы: от текста до живого процесса 🛤️

## Урок 1.1. Полная цепочка

🧠 В основном курсе мы коротко видели «компилятор → линкер → .exe». Теперь — полная картина. Программа проходит **пять станций**:

```
  твой_код.cpp
       │
  ① ПРЕПРОЦЕССОР   раскрывает #include / #define -> чистый текст C++
       ▼
  ② КОМПИЛЯТОР     C++ -> ассемблер -> машинный код.
       │           Получается ОБЪЕКТНЫЙ файл (.o / .obj) — код «с дырками».
       ▼
  ③ ЛИНКЕР         склеивает все .obj + библиотеки, заполняет «дырки»,
       │           назначает адреса -> готовый ИСПОЛНЯЕМЫЙ файл (.exe)
       ▼
  ──────────── файл лежит на диске ────────────
       │
  ④ ЗАГРУЗЧИК ОС   при запуске: раскладывает файл в память, подгружает DLL,
       │           готовит стек -> прыгает в точку входа
       ▼
  ⑤ ПРОЦЕСС        живая программа в памяти, выполняется процессором
```

🤖 Станции ①②③ происходят, когда ты жмёшь **«Собрать»** (на твоём компьютере, заранее). Станции ④⑤ — когда программу **запускают** (может быть на другом компьютере, через год). Реверсеру обычно достаётся только результат — файл `.exe` — и он пытается понять, что было на станциях ①②.

## Урок 1.2. Что делает компилятор (подробнее)

🧠 Компилятор переводит **один** файл `.cpp` в **один** объектный файл. Но программа из нескольких файлов, и одни файлы используют функции из других. Компилятор переводит только то, что видит, а на чужие функции оставляет **«дырки»** — пометки «сюда позже подставить адрес функции `X`».

Объектный файл (`.obj`) содержит:
- **машинный код** уже переведённых функций,
- **таблицу символов** — список имён (функций, переменных): что этот файл *даёт* другим и что ему *нужно* от других,
- **релокации** — те самые «дырки», которые потом заполнит линкер.

🕵️ **Реверс-смысл:** таблица символов — золото для реверсера. По именам функций (`check_password`, `connect_to_server`) сразу понятно, что делает программа. Поэтому защитники часто **«вырезают» имена** — это называется **stripped** (зачищенный) бинарник. Тогда реверсеру тяжелее: вместо `check_password` он видит `sub_401A20` (просто адрес).

---

## Урок 1.3. Что делает линкер (подробнее)

🧠 **Линкер** (компоновщик) — это сборщик. Он:
1. Берёт все объектные файлы.
2. **Разрешает символы** — заполняет «дырки»: нашёл, где функция `X` определена, и подставил её адрес.
3. Подключает **библиотеки** — чужой готовый код.
4. Складывает всё в один файл и раскладывает по **секциям** (про них — Часть 2).

🧠 Библиотеки бывают двух видов — это **очень важно** для реверса:

| | Статическая библиотека | Динамическая библиотека |
|---|---|---|
| Файл | `.lib` (Win) / `.a` (Unix) | `.dll` (Win) / `.so` (Linux) / `.dylib` (mac) |
| Когда подключается | **на сборке**, код вшивается внутрь `.exe` | **при запуске**, отдельным файлом рядом |
| Размер `.exe` | больше (всё внутри) | меньше (код снаружи) |
| Реверс-смысл | весь код — внутри файла | программа **импортирует** функции из DLL → видно **список импортов** 🕵️ |

🤖 Самое вкусное во второй колонке: если программа использует динамические библиотеки, у неё есть **таблица импортов** — список «какие чужие функции мне нужны». Открыв её, реверсер видит намерения программы. Это центральная тема Части 5 (WinAPI).

## Урок 1.4. Загрузчик ОС: рождение процесса

🧠 Файл `.exe` на диске — ещё **не** программа. Это как чертёж. Когда ты его запускаешь, в дело вступает **загрузчик операционной системы**. Он:

1. Читает заголовки файла — узнаёт, сколько памяти нужно и где какие секции.
2. **Отображает секции в память** (код, данные…).
3. Находит и подгружает нужные **DLL**, заполняет таблицу импортов реальными адресами функций.
4. Готовит **стек** и кучу.
5. Прыгает в **точку входа** — это ещё не твой `main`, а сначала маленький «стартовый код» (C runtime), который настраивает окружение и **потом** уже вызывает `main`.

🤖 Запомни: файл на диске → загрузчик → **процесс** в памяти. Реверсеры работают и так и так: **статический анализ** (изучают файл на диске, не запуская) и **динамический анализ** (запускают в безопасной песочнице и смотрят процесс живьём). Отладчик из Урока 5.3 основного курса — это динамический анализ.

🎯 **Попробуй сам (без кода):** объясни своими словами разницу между «файл .exe на диске» и «процесс в памяти». Если получилось — ты понял ключевую идею Части 1.

# Часть 2. Анатомия исполняемого файла 🧬

## Урок 2.1. Файл — это не только код

🧠 Внутри `.exe` не просто свалка машинных команд. Это аккуратно устроенный файл: сначала **заголовки** (служебная информация: где точка входа, какие нужны DLL, сколько памяти), потом **таблица секций**, потом сами **секции** с кодом и данными.

Форматы исполняемых файлов у разных ОС разные:

| ОС | Формат | Расширения | Чем смотреть (реверс) |
|---|---|---|---|
| Windows | **PE** (Portable Executable) | `.exe`, `.dll` | `dumpbin`, `objdump`, PE-bear, Ghidra |
| macOS | **Mach-O** | (без расширения), `.dylib` | `otool`, `nm`, Hopper, Ghidra |
| Linux | **ELF** | (без расширения), `.so` | `readelf`, `objdump`, Ghidra |

🤖 Ты на Windows, поэтому твои программы — это **PE**. Но реверсер должен узнавать все три «в лицо». У всех идея одна: заголовки + секции.

---

## Урок 2.2. Секции внутри файла

🧠 Внутри файла данные разложены по **секциям** — у каждой своё назначение. Главные:

| Секция (PE имя) | Что хранит | Права |
|---|---|---|
| `.text` | машинный код (инструкции) | читать + **выполнять** |
| `.rdata` / `.rodata` | **константы и строки-литералы** | только читать |
| `.data` | глобальные переменные **с** начальным значением | читать + писать |
| `.bss` | глобальные переменные **без** значения (обнуляются) | читать + писать |
| `.idata` | таблица импортов (какие DLL/функции нужны) | только читать |

⚠️ Интересный факт про `.bss`: переменные без начального значения **не хранятся в файле** — записан лишь их размер. Зачем тащить в файле тысячу нулей? Загрузчик сам обнулит этот кусок в памяти при запуске. Так файл получается меньше.

---

## Урок 2.3. 🛠 Твой первый инструмент статического анализа: `strings`

🧠 Помнишь секцию `.rodata` со строками-литералами? Все текстовые строки из твоего кода лежат в готовом `.exe` **как есть, открытым текстом**. Их можно вытащить, даже не запуская программу. Инструмент называется `strings`.

🛠 Эксперимент «найди секрет в файле»:
1. Собери вот такую программу (это наш мини-CrackMe, но цель — не игра, а исследование файла):
   ```cpp
   #include <iostream>
   #include <string>

   int main() {
       std::string vvod;
       std::cout << "Vvedi klyuch dostupa: ";
       std::cin >> vvod;
       if (vvod == "ADMIN_7788") {        // секрет «зашит» прямо в код
           std::cout << "Dostup otkryt\n";
       } else {
           std::cout << "Otkazano\n";
       }
   }
   ```
2. Собери (`Ctrl+B`). Найди готовый `.exe` (Qt Creator показывает путь сборки, обычно папка вида `build-...`).
3. Прогони по нему `strings`. В Qt Creator есть консоль; если рядом стоит MinGW, команда:
   ```
   strings tvoya_programma.exe
   ```
   (на Windows можно поставить Sysinternals `strings.exe`, или использовать `findstr`/Ghidra).
4. Среди вывода ты найдёшь строчку **`ADMIN_7788`** — прямо в файле! Ты «взломал» программу, ни разу её не запустив. 🕵️

✅ **Правило Боба (важное для защиты):** **Никогда не зашивай секреты (пароли, ключи, адреса серверов) прямо в код строками.** Они лежат в `.rodata` открытым текстом, и `strings` находит их за секунду. Это работа белого хакера — заметить такую дыру и сказать «так нельзя». Как делать правильно (хеши, отдельные защищённые хранилища) — разберём, когда будем дорабатывать курс.

# Часть 3. Карта памяти живого процесса 🗺️ (главная глава)

## Урок 3.1. Общая карта

🧠 Когда программа запущена, её память выглядит примерно так (адреса снизу — маленькие, сверху — большие):

```
  БОЛЬШИЕ адреса
  ┌───────────────────────────┐
  │   СТЕК (stack)            │  локальные переменные, аргументы,
  │            ↓ растёт вниз  │  адреса возврата функций
  │                           │
  │      (свободно)           │
  │                           │
  │            ↑ растёт вверх │
  │   КУЧА (heap)            │  new / make_unique / malloc
  ├───────────────────────────┤
  │   .bss                    │  глобальные без значения (нули)
  ├───────────────────────────┤
  │   .data                   │  глобальные со значением
  ├───────────────────────────┤
  │   .rodata                 │  константы, строки-литералы
  ├───────────────────────────┤
  │   .text (КОД)             │  машинные инструкции программы
  └───────────────────────────┘
  МАЛЕНЬКИЕ адреса
```

🤖 Обрати внимание на главную деталь: **стек растёт вниз, куча растёт вверх**, навстречу друг другу, а между ними — свободное место. Каждую секцию разберём отдельно — и сразу с реверс-смыслом.

---

## Урок 3.2. Секции по очереди (с точки зрения хакера)

🧠 **КОД (`.text`)** — машинные инструкции. Помечен «только читать + выполнять» (писать нельзя — защита от того, чтобы программа сама себя переписала). Это то, что реверсер **дизассемблирует** (Урок 5.4 основного курса). Здесь живёт вся логика: проверки паролей, шифрование, сетевые вызовы.

🧠 **`.rodata`** — константы и строки. 🕵️ Здесь `strings` находит пароли, URL, IP-адреса, сообщения. Первое, куда смотрит аналитик.

🧠 **`.data`** — глобальные переменные с начальным значением (`int g_port = 443;` на глобальном уровне). Хранятся в файле, читаются и меняются при работе.

🧠 **`.bss`** — глобальные без значения (`char g_buffer[1024];` глобально). В файле только размер, при запуске обнуляются.

🧠 **КУЧА (heap)** 🔓 — «большой склад», откуда берут память по запросу (`new`, `make_unique`, `malloc`). Программа сама решает, когда взять и вернуть. Реверс-смысл: ошибки работы с кучей — это целый класс уязвимостей: *use-after-free* (использовать память после возврата), *heap overflow* (запись за край). Помнишь **RAII** из углублённого файла? Он как раз спасает от половины таких багов, потому что возвращает память автоматически.

🧠 **СТЕК (stack)** 🔓🕵️ — самое «горячее» место для хакера. Здесь лежат локальные переменные, аргументы функций и, главное, **адреса возврата**. Разберём его отдельно — это сердце классического реверса.

---

## Урок 3.3. 🛠 Увидеть карту памяти своими глазами

🧠 Давай напечатаем адреса переменных из разных секций и убедимся, что карта настоящая.

```cpp
#include <iostream>
#include <memory>

int   g_init   = 1337;                 // .data   (инициализированная глобальная)
int   g_zero;                          // .bss    (неинициализированная глобальная)
const char* g_secret = "S3CRET_KEY";   // строка-литерал -> .rodata

void target() {                        // машинный код функции -> .text
    int local = 0;                          // стек
    auto onheap = std::make_unique<int>(0); // куча

    std::cout << ".text  (kod target): " << reinterpret_cast<void*>(&target)   << "\n";
    std::cout << ".rodata(stroka)    : " << static_cast<const void*>(g_secret) << "\n";
    std::cout << ".data  (g_init)    : " << static_cast<void*>(&g_init)        << "\n";
    std::cout << ".bss   (g_zero)    : " << static_cast<void*>(&g_zero)        << "\n";
    std::cout << "heap   (new)       : " << static_cast<void*>(onheap.get())   << "\n";
    std::cout << "stack  (local)     : " << static_cast<void*>(&local)         << "\n";
}

int main() { target(); }
```

🤖 Запусти. Ты увидишь шесть адресов (в hex!). Посмотри на них: адреса кода/`.rodata`/`.data` — **маленькие**, адрес стека — **большой**, куча — где-то посередине-сверху. Это та самая карта из Урока 3.1, только настоящая, для твоей программы.

⚠️ Тонкость про функцию: `&target` я привёл к `void*` через `reinterpret_cast`. Если этого не сделать, `std::cout` подумает, что это «да/нет», и напечатает `1`. Это классическая ловушка C++ — знать полезно.

🎯 **Попробуй сам:** запусти программу **два раза**. Заметил, что адреса **меняются** между запусками? Это не баг — это защита **ASLR** (про неё в Уроке 3.5). Операционная система специально каждый раз кладёт программу по новым адресам, чтобы хакеру было труднее.

## Урок 3.4. Стек подробнее: как работают вызовы функций

🧠 Помнишь структуру данных **стек** (стопка коробок, LIFO) из углублённого файла? Процессор использует её **постоянно**, для каждого вызова функции.

Когда функция `A` вызывает функцию `B`:
1. На стек кладётся **адрес возврата** — «куда вернуться в `A`, когда `B` закончит».
2. Сверху кладётся **кадр стека** (*stack frame*) функции `B`: её локальные переменные и аргументы.
3. `B` работает в своём кадре.
4. Когда `B` делает `return` — её кадр снимается со стека, процессор читает **адрес возврата** и прыгает обратно в `A`.

```
        стек во время вызова B():
   ┌─────────────────────────┐ ← верх стека
   │ локальные переменные B  │
   │ аргументы B             │
   │ АДРЕС ВОЗВРАТА в A  ◄────┼── сюда вернёмся после B
   ├─────────────────────────┤
   │ кадр функции A          │
   └─────────────────────────┘
```

🕵️ **На ассемблере** это команды `call` (вызвать: положить адрес возврата и прыгнуть) и `ret` (вернуться: снять адрес возврата и прыгнуть назад). Когда реверсер читает дизассемблер, он постоянно видит `call` и `ret` — это и есть путешествие по стеку. Вот почему «стек» — слово №1 в реверсе.

## Урок 3.5. 🔓 Переполнение буфера и защиты (концептуально)

🧠 Теперь — самая знаменитая уязвимость в истории, на уровне идеи (без всякого вредоносного кода).

Представь, что на стеке есть маленькая коробочка-буфер на 8 символов, а **сразу под ней** лежит адрес возврата (Урок 3.4). Если программа невнимательно копирует в буфер строку длиной 100 символов — лишние символы **затирают** то, что рядом, в том числе **адрес возврата**.

```
   до:                    после переполнения:
   [ буфер 8 байт ]       [ AAAAAAAA ]
   [ адрес возврата]  →   [ AAAAAAAA ]  ← адрес возврата ИСПОРЧЕН
```

🔓 Если адрес возврата затёрт, то после `return` процессор прыгнет **не туда** — программа упадёт, а в плохом случае злоумышленник может направить её на свой код. Это называется **переполнение буфера** (*buffer overflow*) — на нём держались тысячи взломов 1990–2000-х.

🛡️ Сегодня от этого защищаются (и тебе как белому хакеру это надо знать):
- **Stack canary** («канарейка») — секретное число кладут перед адресом возврата. Перед `return` проверяют: если число изменилось — значит, был перелив, программа аварийно завершается.
- **ASLR** — случайные адреса при каждом запуске (ты видел это в Уроке 3.3). Хакер не знает, куда прыгать.
- **DEP / NX** — память с данными помечена «нельзя выполнять». Даже если хакер залил свой код в буфер — процессор откажется его запускать.

✅ **Правило Боба:** переполнения возникают из **невнимательной работы с размерами**. Поэтому в этом курсе мы используем `std::string` и `std::vector` с `.at()` (Урок F.2 углублённого файла) — они **сами следят за границами**. Понимать переполнение нужно, чтобы **писать безопасный код и находить такие дыры в чужом** (с разрешения!), а не чтобы ломать.

# Часть 4. Работа с сетями (с уклоном на реверс) 🌐

## Урок 4.1. Базовые понятия сети

🧠 Чтобы реверсить сетевые программы, надо понять, как они вообще общаются.

- **IP-адрес** — адрес компьютера в сети. Как адрес дома: `93.184.216.34`.
- **Порт** — номер «двери/квартиры» в этом доме. У веба порт `80` (HTTP) и `443` (HTTPS), у почты свои. Адрес + порт = точное место.
- **Клиент и сервер** — клиент звонит (`connect`), сервер ждёт звонков (`listen`/`accept`). Браузер — клиент, сайт — сервер.
- **TCP и UDP** — два способа общения. TCP — надёжный «телефонный разговор» (всё дойдёт по порядку). UDP — быстрые «открытки» (можно потерять, зато быстро).
- **Сокет** (*socket*) — «трубка», через которую программа говорит с сетью. Программа создаёт сокет, подключается и шлёт/принимает данные.

---

## Урок 4.2. 🕵️ Зачем это реверсеру и аналитику малвари

🧠 Один из главных вопросов при анализе подозрительной программы: **«а в интернет она лезет? и куда?»**

- Программа открывает сокет и **отправляет** данные на незнакомый IP → возможно, она «звонит домой» (отправляет украденное хозяину). Это называют *C2 / command-and-control* — связь вируса с хозяином.
- Программа что-то **скачивает** → возможно, подгружает дополнительный вредоносный модуль.

🤖 Поэтому аналитик смотрит:
1. **Строки** в `.rodata` — там часто видны IP-адреса и URL открытым текстом (привет, `strings` из Урока 2.3!).
2. **Импорты** — какие сетевые функции программа вызывает (Часть 5).
3. **Живой трафик** — запускают программу в безопасной лаборатории и смотрят, с кем она говорит (инструмент **Wireshark**, команда `netstat`).

⚠️ Снифать (перехватывать) можно только **свой** трафик в своей лаборатории. Чужой — нельзя.

---

## Урок 4.3. 🛠 Как программа выходит в сеть (минимальный пример)

🧠 Сама по себе C++-программа в сеть не ходит — она **просит операционную систему** через API. На Windows это **Winsock** (`ws2_32.dll`). Вот минимальный TCP-клиент. Не пугайся длины — главное, обрати внимание на **названия функций**: ровно их аналитик и увидит в импортах.

```cpp
#include <winsock2.h>
#include <ws2tcpip.h>
#include <iostream>
// ⚠️ при сборке нужна сетевая библиотека. В .pro добавь:  LIBS += -lws2_32

int main() {
    WSADATA wsa;
    WSAStartup(MAKEWORD(2, 2), &wsa);                      // 1. включить Winsock

    SOCKET s = socket(AF_INET, SOCK_STREAM, IPPROTO_TCP);  // 2. создать TCP-сокет

    sockaddr_in addr{};
    addr.sin_family = AF_INET;
    addr.sin_port   = htons(80);                           // порт 80 (веб)
    inet_pton(AF_INET, "93.184.216.34", &addr.sin_addr);   // IP сервера (пример)

    if (connect(s, (sockaddr*)&addr, sizeof(addr)) == 0)   // 3. подключиться
        std::cout << "Podklyuchilis k serveru!\n";
    else
        std::cout << "Soedinenie ne udalos.\n";

    closesocket(s);                                        // 4. закрыть сокет
    WSACleanup();                                          // 5. выключить Winsock
}
```

🤖 Заметил? `WSAStartup` и `socket` — это «взять ресурс», а `closesocket` и `WSACleanup` — «вернуть». Это снова **RAII-ситуация**: правильнее обернуть сокет в объект, который сам закроется в деструкторе. Здесь я написал «вручную» для наглядности, но в правильном коде сокет — это RAII-объект.

🕵️ **Реверс-смысл:** запомни этот набор функций — `WSAStartup`, `socket`, `connect`, `send`, `recv`, `closesocket`. Если аналитик видит их в импортах программы — программа **точно работает с сетью**. О чём именно она «говорит» — покажут строки (адрес) и трафик.

# Часть 5. WinAPI с уклоном на реверс 🪟

## Урок 5.1. Что такое WinAPI

🧠 **WinAPI** — это огромный набор функций, которые Windows даёт программам: показать окно, открыть файл, выделить память, запустить процесс, выйти в сеть. Эти функции живут в системных **DLL**:

| DLL | За что отвечает |
|---|---|
| `kernel32.dll` | память, файлы, процессы (ядро системы) |
| `user32.dll` | окна, кнопки, сообщения, ввод |
| `advapi32.dll` | реестр, права, службы, крипта |
| `ws2_32.dll` | сеть (сокеты, Winsock) |

🤖 Твоя программа не умеет сама рисовать окно — она **просит** `user32.dll` это сделать. И вот тут включается магия реверса: список «кого программа просит» (таблица импортов) виден **снаружи**.

---

## Урок 5.2. 🛠 Первый вызов WinAPI

🧠 Самый известный первый пример — всплывающее окошко.

```cpp
#include <windows.h>
// ⚠️ если линкер ругается на MessageBoxW — в .pro добавь:  LIBS += -luser32

int main() {
    MessageBoxW(nullptr, L"Privet ot WinAPI", L"Bob", MB_OK | MB_ICONINFORMATION);
    return 0;
}
```

🤖 `MessageBoxW` — функция из `user32.dll`. Запусти — выскочит настоящее окно Windows. Ты только что напрямую дёрнул операционную систему.

🕵️ Откуда буква `W` в конце? Многие WinAPI-функции существуют в двух версиях:
- `...A` — старые, обычные строки (ANSI), например `MessageBoxA`.
- `...W` — современные, широкие строки Unicode (`L"..."`), например `MessageBoxW`.

Реверсеру это важно: текст из `W`-функций лежит в файле как **широкие** строки (с нулевыми байтами между буквами), и обычный `strings` их по умолчанию не видит — нужен режим «искать Unicode» (`strings -el`). Маленькая хитрость, на которой новички спотыкаются.

---

## Урок 5.3. 🕵️ Главное: импорты раскрывают намерения

🧠 Вот ради чего вся глава. Аналитик берёт незнакомый `.exe`, смотрит **список импортируемых WinAPI-функций** — и уже понимает, что программа умеет. Это называется **триаж по импортам**.

| Что видит в импортах | О чём это говорит |
|---|---|
| `CreateFileW`, `ReadFile`, `WriteFile` | работает с файлами |
| `RegOpenKeyEx`, `RegSetValueEx` | лезет в реестр (часто — чтобы прописаться в автозагрузку 🚩) |
| `socket`, `connect`, `send` (ws2_32) | выходит в сеть 🌐 |
| `InternetOpen`, `WinHttpConnect` | качает что-то по HTTP |
| `CreateProcess`, `ShellExecute` | запускает другие программы |
| `OpenProcess`, `VirtualAllocEx`, `WriteProcessMemory`, `CreateRemoteThread` | лезет в **чужой** процесс — классический признак внедрения кода 🚩🚩 |
| `CryptEncrypt`, `CryptGenKey` | шифрует данные (у вируса-шифровальщика — красный флаг 🚩) |
| `IsDebuggerPresent` | проверяет, не изучают ли её в отладчике (анти-реверс 🚩) |

🤖 Видишь, как много можно понять, **даже не запуская** программу? Это и есть сила статического анализа. Последние строки таблицы (`WriteProcessMemory`, `CreateRemoteThread`, `IsDebuggerPresent`) — то, что аналитик малвари распознаёт мгновенно. ⚠️ Мы здесь учимся их **узнавать и понимать** — это работа защитника. Писать вредоносный код мы не будем; наша цель — читать и анализировать.

## Урок 5.4. 🛠 Посмотреть импорты своей программы

Ты увидишь среди импортов `user32.dll` → `MessageBoxW`. Поздравляю — ты прочитал «намерение» программы снаружи. 🕵️

# Часть 6. macOS API с уклоном на реверс 🍎

> ⚠️ Ты работаешь на Windows, поэтому эта часть — **для кругозора**. Но хороший реверсер узнаёт программы для любой системы. Здесь — короткое сравнение «как то же самое устроено на Mac».

## Урок 6.1. Из чего состоит macOS API

🧠 У macOS два слоя:
1. **POSIX / BSD** — общий «юниксовый» фундамент (как у Linux). Файлы: `open`/`read`/`write`. Сеть: **обычные BSD-сокеты** (`socket`/`connect` — почти как Winsock, но без `WSAStartup`).
2. **Apple Frameworks** — высокоуровневые наборы: `Foundation`, `Cocoa` (окна, кнопки). Они написаны на Objective-C/Swift.

🤖 Аналогии для реверсера:

| Задача | Windows (WinAPI) | macOS |
|---|---|---|
| Показать окно-сообщение | `MessageBoxW` (user32) | `NSAlert` (AppKit) |
| Открыть файл | `CreateFileW` (kernel32) | `open()` (POSIX) |
| Сетевой сокет | `socket` + `WSAStartup` (ws2_32) | `socket()` (POSIX, без старта) |
| Динамическая библиотека | `.dll` | `.dylib` |
| Формат файла | PE | Mach-O |

---

## Урок 6.2. Формат Mach-O и инструменты

🧠 Программы macOS — это **Mach-O**. Вместо «секций» главную роль играют **сегменты** (`__TEXT` — код и константы, `__DATA` — данные) и **load commands** (команды загрузчика). Динамический загрузчик называется **dyld**.

🛠 Инструменты реверса на macOS (аналоги наших):

| Инструмент | Что делает | Аналог на Windows |
|---|---|---|
| `otool -L` | какие `.dylib` использует программа | `objdump -p` / Dependency Walker |
| `otool -tv` | дизассемблировать код | дизассемблер в Ghidra |
| `nm` | список символов (имён) | таблица символов |
| `dtruss` / `dtrace` | следить за системными вызовами вживую | API Monitor / Process Monitor |
| `lldb` | отладчик | отладчик Qt / x64dbg |
| Hopper, Ghidra | мощный дизассемблер/декомпилятор | то же |

🤖 Главная мысль: **C++-код переносим** между Windows и Mac, а вот в сеть и к файлам он лезет через **разные API**. Реверсер учится читать импорты любой системы — принцип везде один: «по тому, что программа просит у ОС, видно, что она делает».

🎯 **Попробуй сам (на бумаге):** возьми список из Урока 5.3 (импорты Windows) и попробуй для трёх строк придумать, как то же самое называлось бы на macOS (подсказки — в таблице выше).

## 📋 Шпаргалки

**Секции памяти → реверс-смысл:**

| Секция | Хранит | Что ищет тут хакер |
|---|---|---|
| `.text` | код | дизассемблирует логику (проверки, шифрование) |

**Импорты-«флаги» (быстрый триаж):**

| Группа | Функции | Сигнал |
|---|---|---|
| Файлы | CreateFile/ReadFile/WriteFile | работа с диском |
| Реестр | RegSetValueEx | автозагрузка/persistence 🚩 |
| Сеть | socket/connect/send, WinHttp | связь с сетью 🌐 |
| Чужой процесс | WriteProcessMemory/CreateRemoteThread | внедрение кода 🚩🚩 |
| Крипта | CryptEncrypt | шифровальщик? 🚩 |
| Анти-реверс | IsDebuggerPresent | прячется от анализа 🚩 |

## 📖 Словарик (новые слова)

| Слово | Простыми словами |
|---|---|
| **Объектный файл** | Результат компиляции одного `.cpp`, код «с дырками» |
| **Символ** | Имя функции или переменной в таблице |
| **Релокация** | «Дырка», которую заполняет линкер |
| **Stripped** | Бинарник с вырезанными именами — реверсить труднее |
| **Статическая/динамическая библиотека** | Код, вшитый внутрь / подключаемый отдельным файлом (DLL) |
| **Загрузчик** | Часть ОС, раскладывающая `.exe` в память при запуске |
| **Точка входа** | Адрес, с которого начинается выполнение |
| **PE / Mach-O / ELF** | Форматы исполняемых файлов: Windows / macOS / Linux |
| **Секция / сегмент** | Часть файла/памяти со своим назначением (`.text`, `.data`…) |
| **Куча (heap)** | Память «по запросу» (new/malloc) |
| **Стек (stack)** | Память функций: локальные + адреса возврата |
| **Кадр стека (stack frame)** | Кусок стека одного вызова функции |
| **Адрес возврата** | Куда вернуться после `return` |
| **Buffer overflow** | Запись за край буфера, затирающая соседнюю память |
| **ASLR / DEP / canary** | Защиты от переполнений |
| **Сокет / порт / IP** | Трубка для сети / номер двери / адрес компьютера |
| **Импорт / таблица импортов** | Список чужих функций, которые просит программа |
| **WinAPI / DLL** | Функции Windows / файлы, где они живут |
| **Триаж по импортам** | Понять, что делает программа, по списку её вызовов |
