# Часть 1. Что такое асинхронность ⏳

## Урок 1.1. Синхронно против асинхронно

🧠 Представь оператора у станка, который долго что-то делает (например, печатает деталь 5 минут):

- **Синхронно (блокирующе):** оператор нажал «старт» и **стоит, смотрит** на станок все 5 минут, ничего больше не делая. Закончилось — пошёл дальше.
- **Асинхронно (не блокирующе):** оператор нажал «старт» и **ушёл к другим станкам**. Когда деталь готова — станок **сигналит** (лампочка/звонок 🔔), и оператор возвращается забрать её.

🤖 Бытовой пример той же идеи: **стиральная машина**. Ты не стоишь над ней весь цикл — ты занимаешься своими делами, а она **пикнет**, когда закончит. Это и есть асинхронность: *запустил долгое дело и не простаиваешь в ожидании, а получаешь сигнал по готовности.*

✅ Ключевое слово: **не блокировать** (не «замораживать» программу на ожидании).

---

## Урок 1.2. Чем асинхронность отличается от потоков 🧵 vs ⏳

🧠 Это **самый важный** урок файла. Потоки и асинхронность решают **разные** задачи, хотя оба про «несколько дел».

- **Потоки** (файл про многозадачность) = **больше работников**. Берём задачу и делим её между несколькими рабочими/ядрами, чтобы считать **быстрее**. Это про **вычисления**.
- **Асинхронность** = **один работник не простаивает**. Пока долгое дело «варится» само (его делает сеть, диск, другое устройство), работник занимается другим. Это про **ожидание**.

🏭 Сравни:
- Нужно **посчитать** что-то тяжёлое → нанять **много рабочих** (потоки): каждый считает свой кусок.
- Нужно **дождаться** ответа (станок печатает, сеть отвечает) → **один** рабочий запускает дело и идёт к другим станкам, не стоя без дела (асинхронность).

🤖 И главное: их часто **сочетают**. Асинхронно ждём данные из сети, а когда пришли — отдаём их потокам, чтобы быстро обработать.

| | Асинхронность ⏳ | Потоки 🧵 |
|---|---|---|
| Для чего | **ожидание** (сеть, диск, БД, таймер) | **вычисления** (нагрузка на процессор) |
| Сколько работников | часто **один** (цикл событий) | **много** |
| Что экономит | не простаивает в ожидании | использует все ядра |
| Ускоряет вычисления? | **нет** | **да** |
| Главная сложность | колбэки/сигналы, нелинейность | гонки, мьютексы |

## Урок 1.3. Где «теряется» время на ожидании

🧠 Асинхронность полезна там, где программа **ждёт что-то медленное**, а сам процессор в это время простаивает:

- ответ из **сети** (запрос на сайт, к серверу);
- чтение/запись **файла** на диске;
- запрос к **базе данных**;
- **таймер** («сделай через 5 секунд»);
- ответ **пользователя** (нажатие кнопки).

🤖 Во всех этих случаях процессор мог бы заниматься другим, пока ждёт. Асинхронность даёт ему эту возможность.

# Часть 2. Где и зачем применять (а где нет) 🧭

## ✅ Асинхронность даёт максимум пользы, когда:

1. **Много ожидания ввода-вывода** (I/O): сеть, диск, БД. Процессор не должен стоять и ждать.
2. **Нельзя «заморозить» окно.** Пока ждём ответ из сети — окно должно оставаться живым (в Qt это делается асинхронно, **даже без отдельных потоков**).
3. **Очень много одновременных ожиданий.** Сервер на 10000 соединений или сканер на 1000 портов: один **цикл событий** тянет все ожидания сразу — это дёшево. А 10000 потоков — дорого и тяжело.

🤖 Помнишь обещание из файла про сканер — «сделать быстрый режим»? Вот он: асинхронность позволяет одному потоку «жонглировать» тысячами ожиданий.

## ⚠️ Асинхронность НЕ поможет, когда:

- Работа — это **тяжёлые вычисления** (не ожидание, а реальный счёт). Тут асинхронность одна **не ускорит** ничего — процессор и так занят. Нужны **потоки/ядра** (параллелизм).

✅ **Правило Боба:** ждёшь медленное (сеть/диск) → **асинхронность**. Считаешь тяжёлое → **потоки**. Часто — **и то, и другое** вместе.

# Часть 3. Как это работает внутри ⚙️

## Урок 3.1. Цикл событий — диспетчер уведомлений

🧠 Помнишь `app.exec()` из файла про Qt — **цикл событий**? Это и есть «мотор» асинхронности. Представь **диспетчера с рацией**:

1. Ты говоришь: «начни загрузку; когда готово — сообщи мне».
2. Диспетчер (цикл событий) запускает дело и **сразу свободен** для следующих просьб.
3. Когда дело готово, по рации приходит «готово!» 🔔 — и диспетчер вызывает твою функцию-реакцию.

🤖 Один диспетчер так обслуживает **сотни** дел: он никогда не стоит и не ждёт — он только раздаёт старты и реагирует на «готово».

## Урок 3.2. Future — «талон на результат» 🎫

🧠 Второй способ думать про асинхронность — **обещание/будущее** (*future*). Ты запускаешь долгое дело и сразу получаешь **талон**: «результата ещё нет, но он будет; вот талон, по нему заберёшь». Идёшь заниматься своими делами, а когда нужен результат — приходишь с талоном.

---

# Часть 4. Способ 1 — `std::future` / `std::async` (стандартный C++) 🎫

🧠 Самый простой способ «запусти в фоне, результат заберу позже»:

```cpp
#include <iostream>
#include <future>

int longTask() {
    // представь, что это долго: запрос в сеть, чтение большого файла...
    return 42;
}

int main() {
    // запускаем СРАЗУ и идём дальше — нам дают «талон» (future):
    std::future<int> ticket = std::async(std::launch::async, longTask);

    std::cout << "Doing other things...\n";   // работаем, пока задача идёт

    int result = ticket.get();   // «забрать по талону» (подождёт, если ещё не готово)
    std::cout << "Result: " << result << "\n";
}
```

🤖 Разбор:
- `std::async(std::launch::async, ...)` — «запусти эту задачу в фоне».
- `std::future<int> ticket` — твой **талон** на будущий `int`.
- `ticket.get()` — «получить по талону». Если результат уже готов — вернёт сразу; если нет — **подождёт**.

🧠 Главная выгода: между запуском и `.get()` ты **успеваешь сделать другую работу**. Ты ждёшь только в самом конце — и только если результат ещё не готов.

🛠 **`.pro`:** `CONFIG += c++17` (на Linux ещё `-pthread`). Заголовок — `<future>`.

🎯 **Попробуй сам:** запусти **две** долгие задачи через `std::async`, получи два талона, а потом забери оба результата. Обрати внимание: они считались **одновременно**, а не одна за другой.

# Часть 5. Способ 2 — асинхронность через сигналы (Qt) 🔔

🧠 В Qt асинхронность обычно строят на **сигналах** + цикле событий. Это идеально для сети и окон. И вот тут — разгадка, **почему сканер был медленным**.

## Было (синхронно, блокирующе) — как в сканере:

```cpp
QTcpSocket socket;
socket.connectToHost(host, port);
if (socket.waitForConnected(200)) {   // ⛔ БЛОКИРУЕТ: стоим и ждём до 200 мс
    // порт открыт
}
```

🤖 `waitForConnected` **замораживает** программу, пока порт не ответит или не выйдет таймаут. Поэтому порты проверялись **по очереди** — медленно.

## Стало (асинхронно) — не ждём, а подписываемся на «готово»:

```cpp
auto socket = new QTcpSocket;

QObject::connect(socket, &QTcpSocket::connected, [=]() {
    qDebug() << "port" << port << "OPEN";
    socket->deleteLater();                 // прибраться
});
QObject::connect(socket, &QTcpSocket::errorOccurred, [=](QAbstractSocket::SocketError) {
    socket->deleteLater();                 // закрыт / нет ответа
});

socket->connectToHost(host, port);   // ⚡ СРАЗУ возвращается, не ждёт!
// ...прямо тут можно запустить ещё хоть 1000 таких — все ждут параллельно
//    в ОДНОМ цикле событий, без тысячи потоков.
```

🤖 Никакого `waitForConnected`. Мы говорим: «подключайся; **когда** подключишься — вот реакция; если ошибка — вот реакция». И сразу запускаем следующий порт. Один поток «жонглирует» всеми ожиданиями — сканер становится **в разы быстрее**.

⚠️ Чтобы сигналы срабатывали, должен крутиться **цикл событий** (`app.exec()` — Часть 2 файла про Qt).

🧠 Другие асинхронные кирпичики Qt: `QNetworkAccessManager` (HTTP — сигнал `finished`), `QTimer` («позови меня через N мс»), а в Qt 6 — `QFuture` с продолжением `.then(...)` и `QFutureWatcher` (превращает «талон» в сигнал).

# Часть 6. Плюсы и минусы ⚖️

## ✅ Плюсы асинхронности

- **Не тратит процессор на ожидание** — пока ждём I/O, делаем другое.
- **Масштаб:** тысячи одновременных ожиданий на одном цикле событий — дёшево (где тысяча потоков была бы дорогой).
- **Не зависает:** окно/программа остаётся отзывчивой во время долгих операций.
- Для I/O часто **проще**, чем куча потоков с мьютексами.

## ⚠️ Минусы асинхронности

- **Не ускоряет вычисления.** Если работа — это счёт, а не ожидание, async один не поможет (нужны потоки/ядра).
- **Код «рвётся» на кусочки.** Логика расползается по колбэкам/сигналам («что после чего») — это называют *callback hell*. Линейный код читать легче.
- **Труднее отлаживать:** порядок завершения операций **непредсказуем**.
- **Сложнее со временем жизни объектов:** в примере выше пришлось `new` + `deleteLater`, потому что сокет должен пережить функцию (с обычным RAII так просто не выйдет).
- Нужен работающий **цикл событий**.

✅ **Правило Боба:** начни с **простого синхронного** кода (его легко понять). Переходи на асинхронность, когда появляется реальная боль: «программа стоит и ждёт» или «нужно много одновременных ожиданий».

# Часть 7. Async или потоки — что выбрать 🧭

| Ситуация | Бери |
|---|---|
| Ждать сеть / диск / базу данных | **асинхронность** ⏳ |
| Не заморозить окно при долгой операции | **асинхронность** (или фоновый поток) |
| Много одновременных соединений/ожиданий | **асинхронность** (цикл событий) |
| Тяжёлые вычисления на все ядра | **потоки/параллелизм** 🧵 |
| Скачать данные, а потом тяжело их обработать | **и то, и другое**: async на загрузку, потоки на обработку |

🤖 Запомни одной фразой: **асинхронность убирает ожидание, потоки добавляют рабочих.** Это не соперники, а напарники.

# Часть 8. 🛠 Практика: быстрый (асинхронный) сканер портов

🧠 Соберём всё вместе и закроем обещание из файла про сканер. Идея «взрослого» сканера:

1. Пробегаем по всем портам в цикле и для **каждого** создаём `QTcpSocket` в **асинхронном** режиме (как в Части 5) — **не** вызывая `waitForConnected`.
2. Подключаем сигналы `connected` (порт открыт) и `errorOccurred` (закрыт/нет ответа).
3. Вызываем `connectToHost` и **сразу** переходим к следующему порту.
4. Все сотни подключений «висят» одновременно в одном цикле событий; по мере ответов срабатывают сигналы.
5. Открытые порты по сигналу можно сразу показывать в окне (если делаешь GUI-версию).

🤖 Результат: вместо «порт за портом по 200 мс» — все порты проверяются **сразу**. Это и есть та самая разница между блокирующим и асинхронным кодом.

🎯 **Попробуй сам (большой):** переделай свой синхронный сканер портов на асинхронный по шагам выше. Сравни время на диапазоне `1..1024`. Не забудь `return app.exec();`, иначе сигналы не сработают.

⚠️ И снова: сканируй **только своё** (см. этическую рамку в файле про сканер).

# 🕵️ Асинхронность и реверс

- Асинхронный код **труднее читать при реверсе**: вместо прямой линии «сделал A, потом B» — россыпь колбэков и **машина состояний**, которой управляет цикл событий. Логику приходится собирать по кусочкам.
- Цикл событий и регистрация колбэков выглядят характерно — опытный аналитик их узнаёт.
- Идея «не блокироваться» используется и в малвари (например, «спящие» задачи по таймеру, чтобы не выдать себя сразу) — аналитик это учитывает. Мы изучаем такое, чтобы **распознавать и защищать**.

## 📖 Словарик

| Слово | Простыми словами |
|---|---|
| **Синхронно (блокирующе)** | Запустил дело и стоишь ждёшь, ничего больше не делая |
| **Асинхронно (не блокирующе)** | Запустил и пошёл дальше; по готовности придёт сигнал |
| **Цикл событий** | «Диспетчер», который раздаёт уведомления «готово» |
| **Future / «талон»** | Обещание результата, который будет позже |
| **`std::async`** | Запустить задачу в фоне и получить future |
| **`.get()`** | Забрать результат по талону (подождёт, если не готов) |
| **Колбэк / сигнал** | Функция-реакция «когда дело готово — сделай это» |
| **Callback hell** | Когда логика расползлась по множеству колбэков |
| **I/O-bound** | Работа, где в основном **ждут** (сеть, диск) |
| **CPU-bound** | Работа, где в основном **считают** (нужны потоки) |
